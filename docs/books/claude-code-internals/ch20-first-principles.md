# 第 20 章 从第一性原理看 LLM Agent 的本质

> 经过 19 章的源码级拆解，最后一章回归本质问题：LLM Agent 到底是什么？它不是 NLP 模型的包装器，不是 API 调用的自动化脚本——它是一种全新的计算范式。这一章从第一性原理出发，追问这个范式的本质。

---

## 20.1 Agent 不是模型，是系统

Claude Code 是一个 50 万行的项目，分布在约 1900 个文件中。其中直接与 LLM API 交互的代码可能不超过 20%。其余 80% 是什么？

```
System Prompt 设计       (10%)   ← 行为规范
工具系统                 (15%)   ← 能力接口
上下文管理               (20%)   ← 记忆与遗忘
多 Agent 编排            (10%)   ← 并行与协作
权限与安全               (10%)   ← 约束与边界
CLI/UI/终端渲染         (15%)   ← 人机交互
测试/监控/日志           (10%)   ← 质量保障
其他                    (10%)   ← 配置、工具链等
```

Agent 的核心不是"如何调用 LLM"——是**如何围绕 LLM 构建一个完整的计算系统**。LLM 是 CPU，但 CPU 不是计算机。

---

## 20.2 不确定性上的确定性

LLM 的输出本质上是概率性的——同一个 prompt，两次调用返回不同内容。但 Agent 系统需要在不确定性之上构建确定性：

| 机制 | 目的 |
|------|------|
| VCR 录制/重放 | 确定性回放，用于调试和验证 |
| Transcript 持久化 | 不可篡改的完整记录 |
| Deterministic UUID | 可追踪的消息关联（tool_use → tool_result 配对基于 UUID） |
| Zod 验证 | 防止非确定性输出进入下游确定性管道 |
| CacheSafeParams | 确保缓存行为可预测 |

Shannon 的信息论区分了信源（source）和信道（channel）——LLM 是噪声信道，Agent 系统的其余部分是在这个信道上构建可靠通信的编码层。

---

## 20.3 引导而非控制

Agent 系统有一条基本设计原则：**用提示词引导行为，用代码强制约束**。

| 用提示词引导 | 用代码强制 |
|-------------|-----------|
| "默认不写注释" | N/A（无法强制） |
| "遇到障碍时找到根因" | N/A（太复杂无法规则化） |
| N/A | `isConcurrencySafe()` → 分区调度 |
| N/A | `maxResultSizeChars` → 截断 |

规则：**能用代码强制的事情用代码，不能的才依赖提示词**。权限检查是代码强制的，代码风格是提示词引导的。分界线的依据是：能否形式化定义"什么是对的"。

---

## 20.4 感知而非执行

工具设计的核心洞察：工具的**信息价值 > 操作价值**。

即使是一个"只执行"的工具（如 `FileWrite`），它的核心价值也是让 Agent **感知到它对世界做了什么**。Agent 通过 `tool_result` 确认"文件已写入"，这个确认比写入本身更重要——因为 Agent 的下一步推理依赖这个确认。

这就是为什么工具结果的处理（`processToolResultBlock`）和权限检查（`canUseTool`）比工具执行本身（`tool.call`）在代码中占更大篇幅——Agent 系统更多地在处理"工具告诉 Agent 什么"而非"工具对世界做了什么"。

---

## 20.5 Agent 的三个本质问题

回到第 1 章开篇的问题。LLM Agent 需要回答三个问题：

```
1. 我被教导做什么？
   → System Prompt（身份、规则、边界）
   → CLAUDE.md（用户偏好、项目约束）
   → Memory（持久化的经验）

2. 我记得自己做了什么？
   → Messages（当前对话）
   → Compact Summary（压缩后的记忆）
   → Transcript（完整归档）
   → Plan（当前计划状态）

3. 我如何与世界交互？
   → Tools（感知和行动接口）
   → Permissions（安全约束）
   → StreamingToolExecutor（时间感知）
   → Mailbox（与其他 Agent 交互）
```

这 20 章的每一章都可以归入这三个问题的某一个。

## 20.6 AI 时代的操作系统

```
传统计算机               LLM Agent 系统
─────────────────────────────────────────────
CPU 执行指令              LLM 执行推理
内存存储数据              上下文窗口存储 token
磁盘持久化                Transcript 持久化
操作系统管理资源          Agent 框架管理上下文
系统调用                  Tool 接口
用户程序                  Prompt / CLAUDE.md
进程隔离                  子 Agent / Swarm
```

最关键的区别不是技术细节——是抽象层级：

- 传统 OS 管理的是**二进制指令和数据**
- Agent 系统管理的是**语义文本和意图**

从指令到意图，这是一代计算范式的跃迁。Claude Code 不是这个跃迁的终点——它是这个跃迁的第一批严肃尝试。

## 全书结语

回到本书开篇的问题：**Claude Code 到底是如何工作的？**

答案不在任何一个文件、任何一个函数中。答案分布在整个系统之中：

- 在 `getSystemPrompt()` 返回的数组里，每行文本都是对 Agent 行为的精确引导
- 在 `while(true)` 循环中，每次迭代都是"思考→行动→观察"的一次执行
- 在五级压缩管线中，每级都是对"记住什么、忘记什么"的一次权衡
- 在 `StreamingToolExecutor` 中，工具与 LLM 的流水线执行是对时间的一次优化
- 在 AgentTool 的参数里，子 Agent 的调用是对能力的一次"窄化授予"
- 在 `CLAUDE.md` 的文件中，用户的偏好是系统行为的最高优先规则

**Agent 是这些部件协同运作的涌现行为**。没有单个组件定义了 Agent（就像没有单个组件定义了操作系统）。

理解 Agent，就是理解这些部件如何连接为一个整体。这 20 章的目标不是回答"每个部件如何工作"——而是揭示这些部件之间的连接关系，让你看到 1900 个文件背后的整体图景。

**从数据库到 Agent**——这条学习曲线之所以自然，是因为两者都在解决**如何在资源约束下可靠地执行复杂计算**。区别只在于：数据库管理的是精确的字节，Agent 管理的是概率性的语义。当你能同时理解这两者时，你就站在了理解下一代计算系统的门槛上。
