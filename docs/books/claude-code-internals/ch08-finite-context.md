# 第 8 章 有限窗口下的推理：问题的本质

> Agent 最大的工程挑战不是"如何调用工具"，而是"如何在有限窗口中持续推理"。这一章从信息论和数学的角度，建立上下文管理问题的完整认知模型。

---

## 8.1 为什么窗口扩展不能解决根本问题

每代 Claude 模型的上下文窗口都在增长：100K → 200K → 500K → 1M tokens。但这不能解决根本问题——因为**任务的"需要记住的信息"总是比窗口更快增长**。

一个 Agent 执行 10 轮推理后：
- 每轮 ~5K tokens（模型输出 + 工具结果）
- 10 轮 = 50K tokens
- 100 轮 = 500K tokens（已超出大多数窗口）

而复杂的软件工程任务很容易超过 100 轮。扩展窗口只是**推迟了压缩的时刻**，不能消除压缩的必要性。

---

## 8.2 信息衰减的数学

设每轮平均新增 token 数为 `R`，窗口大小为 `W`。在轮数 `n` 之后：

```
总 token 数 = R × n
临界轮数 = W / R
```

对于 Claude Code 的典型使用：
- R ≈ 3000-5000 tokens/轮
- W = 200K tokens
- 临界轮数 ≈ 40-66 轮

一个典型的 bug fix 可能需要 10-20 轮，一个 feature 实现可能需要 50-100 轮。这意味着**在一个复杂任务的执行过程中，Agent 一定会碰到上下文瓶颈**。

---

## 8.3 有损压缩的不可回避性

从信息论视角看：

```
Agent 记忆  = 有限容量的信道
任务信息    = 无限增长的信号源
→ 必须有损压缩
```

就像香农的信道编码定理：如果信号源的熵超过信道容量，就不可能无损传输。Agent 面对的任务信息永远超过上下文窗口，所以有损压缩是数学必然。

---

## 8.4 Claude Code 的设计哲学

Claude Code 对这个问题的态度是独特的：

> **压缩不是异常路径，而是基本运行模式**。

大多数 LLM 框架把"压缩"当作最后的救命稻草——只在 OOM 或 413 错误时才触发。Claude Code 把它当作系统的基本操作模式——就像操作系统通过页面置换管理物理内存，Claude Code 通过压缩管线管理上下文窗口。

### 三个基本假设

1. **上下文窗口永远是稀缺资源**：扩展窗口只是提高上限，不改变稀缺本质
2. **压缩应该渐进而非突发**：从轻量到重度的连续策略，而非"要么不压缩，要么全量压缩"
3. **压缩不应打断用户**：用户不应该感知到压缩在发生

这三个假设直接塑造了第 9 章中的五级压缩管线。

---

**与数据库的类比**：上下文窗口 ≈ buffer pool。页面置换（LRU/CLOCK）≈ token 预算管理。checkpoint ≈ compact boundary。WAL ≈ transcript 持久化。这不是比喻——两者都在解决"有限容量 + 无限需求"的问题。
