# 第 17 章 有损压缩下的推理保真度

> 压缩必然丢失信息。丢失的信息对后续推理的影响有多大？这一章从信息论和认知科学的视角，分析有损压缩如何影响 Agent 的推理质量。

---

## 17.1 信息论视角

Agent 记忆可以被建模为一条带噪声的信道：

```
原始对话: X（完整信息源）
压缩后:   Y（通过压缩函数 C 处理后的信号）
恢复后:   X'（LLM 从 Y 中重建的理解）

信息损失 = H(X) - I(X; Y)
推理保真度 = I(X'; 最优决策) / I(X; 最优决策)
```

关键度量不是"保留了多少原始字节"，而是"保留了用于正确决策的关键信息"。

---

## 17.2 压缩级联的信息损失

五级压缩管线中，每级的信息损失不是线性的：

```
Level 1 (Snip):        损失 < 5%    → 只删重复/冗余
Level 2 (Microcompact): 损失 < 2%    → 缓存层语义等价
Level 3 (Collapse):     损失 < 10%   → 折叠但仍可恢复
Level 4 (Auto Compact): 损失 15-30%  → LLM 主动选择保留什么
Level 5 (Reactive):     损失 30-60%  → 强制截断
```

信息损失的非线性增长意味着：**尽可能在低级别完成压缩**不仅是成本优化，也是质量优化。

---

## 17.3 摘要质量的决定因素

Auto Compact 的摘要质量取决于 compact prompt 的设计。双层 `<analysis>` + `<summary>` 结构是基于一个关键洞察：

> LLM 在选择"保留什么"时如果直接输出摘要，它倾向于输出事实而忽略推理过程。

`<analysis>` 强制 LLM 先思考"用户要做什么、关键决策是什么、什么对后续推理最重要"。这个思考过程虽被丢弃，但它产出了更高质量的 `<summary>`。

---

## 17.4 关键信息的识别与保留

哪些信息在压缩中最需要保留？

| 优先级 | 信息类型 | 原因 |
|--------|---------|------|
| 最高 | 用户偏好和约束 | 不可恢复 |
| 高 | Bug 根因分析 | 推理成本高 |
| 高 | 关键文件路径 | 后续操作依赖 |
| 中 | 工具执行结果 | 可重新执行获取 |
| 中 | 对话中的决定和承诺 | 一致性需要 |
| 低 | ROUTINE 工具调用的完整输出 | 模式化的，不包含新信息 |
| 最低 | 错误重试的中间过程 | 只有最终成功路径重要 |

---

## 17.5 与人类记忆的类比

认知心理学区分了三种记忆：

| 记忆类型 | Agent 对应 | 容量 | 持久性 |
|----------|-----------|------|--------|
| 工作记忆 (Working Memory) | 当前上下文窗口 | 有限（~7±2 items / ~200K tokens） | 会话内 |
| 短期记忆 | Compact Summary | 中等 | 到下次压缩 |
| 长期记忆 | Transcript 文件 + Memory 系统 | 理论无限 | 跨会话 |

压缩是将"工作记忆"转化为"短期记忆"（compact summary），以及将"短期记忆"归档为"长期记忆"（transcript）的过程。

## 17.6 从数据库 Query Engine 视角看

```
compaction ≈ merge sort 的归并阶段
  → 将多个 run 合并为一个有序输出
  → 中间数据丢弃，只保留最终结果

boundary ≈ checkpoint
  → 标记"这里之前的 WAL 已合并到数据文件"
  → 崩溃恢复从最近的 checkpoint 开始

information loss ≈ 位图索引的近似计数
  → HyperLogLog 的 trade-off: 用精度换空间
  → VCR 的 trade-off: 用信息损失换上下文空间
```

---

## 本章小结

有损压缩不是缺陷——它是自然。人类的记忆是有损的，数据库的索引是近似的，操作系统的页面回收是有选择的。Agent 的压缩管线只是这个原则在 LLM 上下文管理上的表达。

关键问题是**识别什么必须保留、什么可以丢弃**——这既是 compact prompt 的工程问题，也是对 Agent 推理本质的哲学问题。
