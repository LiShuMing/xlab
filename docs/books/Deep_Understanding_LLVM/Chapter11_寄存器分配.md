# 第11章 寄存器分配：虚拟到物理的映射

## 11.1 寄存器分配问题：NP-完全与启发式

### 11.1.1 LiveInterval：活跃区间分析

寄存器分配的核心概念是活跃区间(Live Interval)——一个虚拟寄存器从定义点到最后一个使用点之间的范围：

```
基本块内的活跃区间:

  v1 = load i32, ptr %p    ; v1定义 ──────────┐
  v2 = add i32 v1, 1       ; v1使用, v2定义 ───┤ v1活跃: [0, 2)
  v3 = mul i32 v2, 4       ; v2使用, v3定义 ───┤ v2活跃: [1, 3)
  store i32 v3, ptr %q     ; v3使用 ───────────┘ v3活跃: [2, 4)

活跃区间表示: [start_slot, end_slot)
  v1: [0, 2)  ← 在slot 0到1之间活跃
  v2: [1, 3)
  v3: [2, 4)

干涉图(Interference Graph):
  两个活跃区间重叠 → 干涉 → 不能分配同一物理寄存器
  v1和v2干涉(slot 1重叠) → 不能共用寄存器
  v1和v3不干涉 → 可以共用寄存器
```

### 11.1.2 溢出代价模型与分配评分

当物理寄存器不足时，必须将某些虚拟寄存器溢出(spill)到栈上：

```
溢出代价计算:
  spill_cost(vreg) = Σ(使用点权重) / 活跃区间长度

  使用点权重:
    循环内使用: ×循环嵌套深度 (循环深度1: ×10, 深度2: ×100)
    热路径使用: ×PGO权重

  示例:
    vreg1: 活跃区间100个slot, 2个使用点(都在循环深度2)
    spill_cost = 200/100 = 2.0

    vreg2: 活跃区间10个slot, 1个使用点(不在循环)
    spill_cost = 1/10 = 0.1

  → 溢出vreg2更划算(代价更低)
```

**体系结构锚点**：溢出的代价不是常数——它取决于内存层级。L1缓存命中(4周期)vs L2(12周期)vs 主存(100+周期)，实际溢出代价可能差25倍。但编译器通常使用简化的代价模型，因为精确的缓存行为在编译期不可预测。

## 11.2 三种分配算法

### 11.2.1 RegAllocFast：快速局部分配

**源码**：`llvm/lib/CodeGen/RegAllocFast.cpp`（1923行）

RegAllocFast用于O0编译和调试构建——追求编译速度而非代码质量：

```
算法: 线性扫描 + 局部分配
  1. 按指令顺序遍历
  2. 对每条指令的虚拟寄存器:
     - 尝试分配一个空闲的物理寄存器
     - 如果没有空闲寄存器, 溢出当前最不活跃的寄存器
  3. 在基本块边界释放所有分配

特点:
  - 不做全局活跃区间分析
  - 不做溢出代价计算
  - 溢出决策基于简单的启发式
  - 编译速度快, 但代码质量差(大量溢出)
```

### 11.2.2 RegAllocGreedy：贪心全局分配

**源码**：`llvm/lib/CodeGen/RegAllocGreedy.cpp`（2995行）

RegAllocGreedy是O2/O3的默认寄存器分配器——它使用全局活跃区间分析和贪心着色算法：

```
算法:
  1. 计算所有虚拟寄存器的活跃区间
  2. 按溢出代价从高到低排序(代价高的优先分配)
  3. 对每个虚拟寄存器:
     a. 尝试分配一个不与已分配区间干涉的物理寄存器
     b. 如果找到 → 分配
     c. 如果没有空闲物理寄存器 → 选择一个已分配的区间溢出
        - 溢出选择: 代价最低的已分配区间
        - 可能溢出当前虚拟寄存器自己(如果它的代价最低)
  4. 处理溢出:
     - 在溢出点插入store(溢出前保存)
     - 在使用点插入load(使用前恢复)
  5. 重新计算受影响的活跃区间, 重新分配

优化:
  - 分割(Splitting): 将长活跃区间在干涉点分割, 减少干涉
  - 重彩(Recoloring): 局部重新着色以减少溢出
  - 区域分配(Region Allocation): 对热路径优先分配
```

### 11.2.3 RegAllocPBQP：分区布尔二次规划

**源码**：`llvm/lib/CodeGen/RegAllocPBQP.cpp`（950行）

PBQP将寄存器分配建模为分区布尔二次规划(Partitioned Boolean Quadratic Programming)问题：

```
PBQP模型:
  变量: 每个虚拟寄存器选择哪个物理寄存器(或溢出)
  代价矩阵: 寄存器分配的代价 + 溢出代价
  干涉约束: 两个干涉的虚拟寄存器不能选同一物理寄存器

求解:
  1. 构建PBQP图(节点=虚拟寄存器, 边=干涉)
  2. 归约: 逐步消除简单节点(度数≤2的节点可以精确求解)
  3. 回代: 从简化的图反向推导分配方案

优势: 比贪心算法更精确(考虑全局最优)
劣势: 编译时间长(大函数可能指数级), 实际很少使用
```

## 11.3 ML驱动的寄存器分配

**源码**：`llvm/lib/CodeGen/MLRegAllocEvictAdvisor.cpp` + `MLRegAllocPriorityAdvisor.cpp`

LLVM正在将ML模型引入寄存器分配的两个关键决策：

```
1. Evict决策: 当没有空闲寄存器时, 溢出哪个已分配的区间?
   传统: 选择spill_cost最低的
   ML: 基于程序特征预测哪种选择导致最少的总溢出

2. Priority决策: 哪个虚拟寄存器优先分配?
   传统: 按spill_cost从高到低
   ML: 基于程序特征预测最优的分配顺序

输入特征:
  - 活跃区间长度/权重
  - 干涉图的结构(度数, 聚类系数)
  - 循环嵌套深度
  - 已分配/溢出的寄存器数

模型: 决策树或小型神经网络(推理延迟要求<1ms)
```

## 11.4 不同体系结构的寄存器压力差异

### 11.4.1 寄存器数量对比

| 架构 | 通用寄存器 | 向量寄存器 | 特殊寄存器 | 总计 |
|------|-----------|-----------|-----------|------|
| X86-64 | 16 GPR (rax-r15) | 16/32 XMM/YMM/ZMM | 6 段寄存器 | ~38-54 |
| AArch64 | 31 GPR (x0-x30) | 32 NEON/SVE | SP/PC/PSTATE | ~65+ |
| RISC-V | 31 GPR (x1-x31) | 32 V (向量扩展) | PC | ~63+ |

**编译器视角的影响**：

```
X86-64 (16 GPR): 寄存器压力最大
  - 16个GPR中, 几个被固定用途占用(rsp, rbp, rdi-rsi调用约定)
  - 实际可用只有~10个
  - 溢出频率高, 需要更复杂的分配策略
  - 向量化时XMM/YMM寄存器同样紧张

AArch64 (31 GPR): 寄存器压力较低
  - 31个GPR, 调用约定只占用少量
  - 实际可用~25个
  - 溢出频率低, 代码质量更高
  - 这是RISC架构的优势之一

RISC-V (31 GPR): 类似AArch64
  - 31个GPR, x0固定为0
  - 寄存器压力低, 但向量扩展的分配策略更复杂
```

**体系结构锚点**：寄存器数量是ISA设计的关键权衡——更多寄存器意味着更低溢出率、更好代码质量，但也意味着更大的上下文切换开销(保存/恢复更多寄存器)。AArch64的31个GPR是ARM对CISC(X86 16个)的竞争优势之一。

## 11.5 本章小结

本章深入了寄存器分配——编译后端最经典的NP完全问题：

1. **活跃区间分析**：LiveInterval编码了虚拟寄存器的生命周期，干涉图决定了哪些虚拟寄存器不能共用物理寄存器。
2. **三种分配算法**：Fast(快速局部)、Greedy(贪心全局, 默认)、PBQP(精确但慢)。
3. **ML驱动的分配**：Evict和Priority决策的机器学习化，从启发式走向学习型。
4. **体系结构差异**：X86的16 GPR vs AArch64/RISC-V的31 GPR，寄存器压力直接影响溢出频率和代码质量。
