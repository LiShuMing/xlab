# 第23章 Agent与编译器：自适应优化的未来

## 23.1 Agent驱动的编译优化

### 23.1.1 编译决策作为MDP

```
将编译优化建模为马尔可夫决策过程(MDP):

状态(State): 程序特征
  - IR结构: 指令数、基本块数、循环嵌套深度
  - 数据流: 活跃变量数、内存访问模式
  - 控制流: 分支概率、循环迭代次数
  - 体系结构: 寄存器压力、缓存行为

动作(Action): 优化选择
  - 内联/不内联某个调用
  - 展开/不展开某个循环
  - 向量化使用哪个宽度
  - 寄存器分配的溢出选择

奖励(Reward): 性能收益
  - 编译后程序的运行时间(负值 → 最小化)
  - 或: 相对于基线的加速比

状态转移(Transition): 优化后的程序
  - 执行一个动作后, IR变化 → 新状态
```

### 23.1.2 AutoTVM / Ansor / CompilerGym的探索

```
AutoTVM (TVM项目):
  自动搜索最优的tensor算子schedule
  - 搜索空间: tiling大小、unroll策略、向量宽度...
  - 搜索算法: XGBoost + 模拟退火
  - 结果: 为每个算子+目标组合找到接近最优的schedule

Ansor (TVM项目):
  AutoTVM的进化版
  - 搜索空间: 自动生成(不依赖手工模板)
  - 搜索算法: 进化搜索 + Sketch生成
  - 结果: 超越AutoTVM的手工模板

CompilerGym (Meta):
  编译优化的强化学习环境
  - 环境: LLVM编译器
  - 动作: 选择Pass序列
  - 奖励: 代码大小/运行时间改善
  - 特点: OpenAI Gym接口, 方便RL研究

  结果:
  - RL agent可以找到比O2/O3更好的Pass序列
  - 但搜索成本高(需要大量编译-执行-反馈循环)
  - 目前仍处于研究阶段, 尚未生产部署
```

## 23.2 程序表征学习

### 23.2.1 IR2Vec / ProGraML / CodeBERT

```
从IR到向量: 编译器的程序嵌入

IR2Vec:
  源码: llvm/lib/Analysis/IR2Vec.cpp
  方法: 基于数据流的程序表征
  - 每条指令编码为向量(操作码+操作数类型+标志)
  - 通过数据流图传播信息
  - 生成函数/基本块级别的嵌入

  用途:
  - 内联决策(MLInlineAdvisor的特征来源之一)
  - 异常检测(识别与已知性能问题相似的代码模式)
  - 代码相似性搜索

ProGraML:
  方法: 基于GNN(图神经网络)的程序表征
  - 将IR构建为图(节点=指令, 边=数据流+控制流)
  - 使用消息传递GNN学习节点/图嵌入
  - 捕获程序的结构语义

CodeBERT / UniXcoder:
  方法: 基于Transformer的代码预训练模型
  - 在大规模代码语料上预训练
  - 理解代码的语义(变量名、注释、结构)
  - 可用于代码生成、补全、搜索

  与编译器的结合:
  - 从自然语言生成IR(第23.3节)
  - 理解优化注释和意图
  - 自动生成优化Pass
```

## 23.3 程序合成的编译器路径

### 23.3.1 从自然语言到IR的生成

```
LLM作为编译前端:

传统: C++ → Clang → AST → LLVM IR
未来: 自然语言 → LLM → MLIR/LLVM IR

实验性探索:
  1. Prompt → LLM → MLIR Linalg:
     "矩阵乘法C=A×B, A是128×256, B是256×512"
     → linalg.matmul ins(%A, %B) outs(%C)

  2. Prompt → LLM → CUDA kernel:
     "实现softmax kernel for GPU"
     → CUDA kernel代码

  3. Prompt → LLM → 优化Pass:
     "写一个Pass, 将连续的add替换为mul×2"
     → InstCombine规则代码

挑战:
  1. 正确性: LLM生成的IR可能不合法(Verifier会拒绝)
  2. 效率: LLM生成的kernel通常不如手工优化
  3. 安全性: 生成恶意代码的风险
  4. 可验证性: 如何确保生成的代码语义正确?
```

### 23.3.2 CIR(MLIR Clang IR) / CIRCT(硬件综合)

```
CIR (Clang IR):
  Clang正在开发基于MLIR的新的中间表示
  路径: C/C++ → CIR(MLIR Dialect) → LLVM IR

  目标:
  - 在MLIR层面表达C++语义(类、模板、异常)
  - 启用MLIR的变换和优化能力
  - 更好的可扩展性(添加新语言特性只需扩展Dialect)

  当前状态: 实验性, 逐步替代Clang的AST→LLVM IR直接生成

CIRCT (Circuit IR Compilers and Tools):
  基于MLIR的硬件综合框架
  路径: Chisel/Verilog/Amaranth → CIRCT Dialects → Verilog/FIRRTL

  核心Dialects:
  - hw: 硬件基础操作(实例化、端口)
  - comb: 组合逻辑(and, or, xor)
  - seq: 时序逻辑(寄存器、时钟)
  - fsm: 有限状态机
  - arith: 算术操作

  编译器与硬件的统一:
  CIRCT证明MLIR不仅适用于软件编译
  硬件综合和软件编译共享基础设施
  → 未来可能出现软硬协同的统一编译框架
```

## 23.4 展望：AI原生编译器

### 23.4.1 学习型代价模型替代手工启发式

```
当前: 手工启发式 → 维护成本高, 难以适应新硬件
未来: 学习型代价模型 → 自动适应, 持续改进

学习型代价模型的挑战:
  1. 训练数据: 需要大量"程序×优化决策→性能"的样本
  2. 泛化性: 新程序/新硬件上的预测精度
  3. 可解释性: 编译器决策需要可解释(否则无法调试)
  4. 编译时间: 模型推理不能显著增加编译时间

可能的解法:
  - 迁移学习: 在一个硬件上训练, 微调到新硬件
  - 在线学习: 编译器持续收集profile, 更新模型
  - 可解释AI: 使用决策树而非黑箱模型
  - 模型蒸馏: 大模型→小模型, 降低推理延迟
```

### 23.4.2 端到端可微编译管线

```
愿景: 编译管线本身可微分, 可以端到端优化

概念:
  源代码 → [可微编译器] → 机器码 → [运行] → 性能

  编译器的每个决策(内联/展开/分配)都是可微的
  → 可以用梯度下降优化编译决策
  → 类似神经网络训练, 但优化的是编译策略

挑战:
  1. 离散决策: 内联/不内联是不可微的 → 需要松弛化
  2. 长依赖链: 前端决策影响后端结果 → 梯度消失/爆炸
  3. 训练成本: 每次前向传播需要完整编译+运行

可能的路径:
  - Gumbel-Softmax松弛化离散决策
  - 强化学习(不依赖梯度)
  - 进化策略(无梯度优化)
```

### 23.4.3 硬件-编译器协同设计

```
当前: 硬件设计 → 编译器适配(被动)
未来: 硬件-编译器协同设计(主动)

协同设计循环:
  1. 编译器分析: 哪些操作是瓶颈? 需要什么新指令?
  2. 硬件设计: 添加新指令/执行单元
  3. 编译器利用: 生成使用新指令的代码
  4. 性能评估: 新指令是否带来收益?
  5. 迭代优化

实例:
  Intel AVX-512 VNNI: 为INT8推理添加指令
  → 编译器自动生成VNNI代码 → 推理性能翻倍

  ARM SVE: 可伸缩向量扩展
  → 编译器生成vscale代码 → 一次编译适配所有宽度

  RISC-V扩展: Zbb/Zbs/Zba位操作扩展
  → 编译器利用新位操作指令 → 减少指令数

  AI加速器(TPU/IE): 专用矩阵乘法单元
  → 编译器生成systolic array代码 → 极高算力利用率

MLIR在协同设计中的角色:
  - 统一IR: 硬件特性用Dialect表达
  - 快速原型: 新Dialect可以快速试验
  - 渐进Lowering: 从高层到低层逐步适配
  - 可验证性: 每层变换都可以验证正确性
```

## 23.5 本章小结

本章展望了编译器的未来方向：

1. **Agent驱动优化**——编译决策作为MDP，AutoTVM/Ansor/CompilerGym的探索。
2. **程序表征学习**——IR2Vec/ProGraML/CodeBERT将IR编码为向量，支持ML驱动的优化。
3. **程序合成**——LLM作为编译前端，CIR/CIRCT将MLIR扩展到C++编译和硬件综合。
4. **AI原生编译器**——学习型代价模型、端到端可微编译、硬件-编译器协同设计。

从三段式哲学到AI原生编译器，LLVM的演进映射了整个编译领域的发展——从手工规则到机器学习，从单一后端到多架构适配，从软件编译到软硬协同。编译器的基础设施价值永不褪色：**无论上层如何变化，IR始终是枢纽，优化始终是核心，硬件适配始终是目标**。
