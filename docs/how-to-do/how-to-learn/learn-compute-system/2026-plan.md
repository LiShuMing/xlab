
# Plan 1
下面给你一份可直接执行的 2026 年学习计划。目标不是“泛泛学过操作系统、编译器、算法、LLM”，而是建立一套可迁移、可复用、可反哺工作的系统认知框架。整体思路是：上半年夯实计算机基础，下半年进入 LLM/AI 系统；每一阶段都要求形成“概念理解—实现观察—工作映射—书面输出”的闭环，避免只停留在阅读层面。

一、年度目标

2026 年底前，完成四个结果。第一，系统掌握操作系统、编译器、算法与数据结构的核心知识，并能用自己的语言解释。第二，能够把这些基础知识映射到数据库内核、查询优化、执行引擎、分布式系统与 AI infra 场景。第三，形成一套稳定的学习与总结机制，使知识持续沉淀，而不是学完即散。第四，下半年切入 LLM/AI 时，不是停留在概念层，而是从系统视角理解模型、推理、并行与 serving。

二、全年节奏

全年分为三个阶段。第一阶段是 1 至 4 月，主题为操作系统与计算机系统基础，重点解决“程序为什么这样运行”。第二阶段是 5 至 8 月，主题为编译器、算法与程序表示，重点解决“程序如何被表达、分析和优化”。第三阶段是 9 至 12 月，主题为 LLM/AI 系统，重点解决“AI 系统如何建立在前述基础之上”。这三个阶段不是彼此孤立，而是逐步递进：操作系统提供资源管理视角，编译器提供表示与变换视角，算法提供问题建模视角，三者共同构成后续理解 LLM/AI 系统的基础。

三、季度计划

Q1（1—3 月）：操作系统与计算机系统观

这一阶段的核心目标，是建立“程序运行时视角”。你需要真正理解进程、线程、上下文切换、地址空间、页表、TLB、虚拟内存、缺页异常、文件系统、page cache、系统调用、锁与并发原语，而不是只会背定义。建议主教材以《Operating Systems: Three Easy Pieces》和 MIT 6.S081 为主，配合《CSAPP》中内存层次、链接、虚拟内存相关章节。你的重点不是把 xv6 每个细节都啃透，而是通过实验理解：为什么需要虚拟内存，为什么 page cache 会影响性能，为什么锁竞争会放大尾延迟，为什么 NUMA 与 cache locality 会决定系统表现。

Q1 的输出要求很明确：至少写 4 篇系统笔记，主题建议为“虚拟内存与页表”“page cache 与 mmap”“线程、锁与调度”“文件系统与 I/O 路径”。每篇都要求带一段“与数据库内核的关系”，例如 page cache 与 buffer pool 的差异，NUMA 对 scan / join 的影响，锁竞争对 pipeline 执行的影响。只有写出这种映射，你才算真正吸收。

Q2（4—6 月）：操作系统深化 + 算法与数据结构

Q2 的目标，是把“系统运行”与“数据组织”结合起来。操作系统方面，继续巩固并发、内存和 I/O；算法与数据结构方面，则不走刷题路线，而是围绕系统工程需要的核心模型来学：数组、链表、哈希表、树、堆、图、并查集、布隆过滤器、跳表、LSM、优先队列、图遍历、动态规划、贪心、分治。推荐材料可以用《Algorithm Design Manual》或《Algorithms》作为主线，但学习方式必须偏工程化：每个结构与算法都要回答三个问题——它解决什么问题，时间/空间/局部性特征如何，数据库或分布式系统中对应的落点是什么。

Q2 的输出要求是 4 篇方法论笔记，建议主题为“哈希与排序在数据库中的位置”“树、索引与查询代价”“布隆过滤器、跳表、LSM 的工程价值”“图与动态规划在优化器中的隐性身影”。同时，至少做两次小型代码练习，不求花哨，可以是自己手写一个简单的 LRU cache、skip list、bloom filter 或小型内存分配模拟器。目的不是做项目，而是通过手写建立直觉。

Q3（7—9 月）：编译器基础与程序表示

这一阶段是你未来触类旁通的关键。很多数据库工程师对编译器感到陌生，其实优化器、表达式重写、增量计算计划变换，与编译器的 IR、规则变换、数据流分析有很强同构性。建议以《Crafting Interpreters》建立直觉，以《Engineering a Compiler》理解正式框架，再辅以 LLVM IR、SSA、简单 pass 的资料。重点不在于完整实现一个编译器，而在于理解词法/语法/AST/IR/SSA/控制流图/数据流分析/优化 pass 这条链条。

Q3 的输出要求是 4 篇“程序变换”笔记，建议主题为“从 AST 到 IR：程序表示为何重要”“SSA 为什么是现代优化的基础”“数据流分析与规则重写”“编译器优化与 SQL 优化器的共性”。同时，建议做一个非常轻量的小练习：用任意熟悉语言，手写一个表达式解析器，支持简单算术表达式转 AST，再做常量折叠。这个练习不复杂，但非常能帮助你体会“表示—分析—变换”的主线。

Q4（10—12 月）：LLM/AI 系统

进入第四阶段时，你不再从“AI 热门名词”出发，而是从系统视角理解模型与基础设施。重点应覆盖四层：模型基础、训练与推理、并行与内存、服务系统。模型基础需要掌握 Transformer、Attention、位置编码、前馈层、训练损失等；系统层重点理解 KV cache、batching、prefill / decode、张量并行、流水并行、数据并行、显存瓶颈、访存瓶颈、vLLM 的 PagedAttention、DeepSpeed / TensorRT-LLM 的定位。推荐方式不是只读科普，而是选少量高质量资料，配合官方文档、论文与系统实现文章一起看。

Q4 的输出要求是 4 篇 AI 系统笔记，建议主题为“Transformer 的计算与内存结构”“推理系统的瓶颈：算力还是带宽”“KV cache、batching 与调度”“vLLM / DeepSpeed / TensorRT-LLM 分别解决什么问题”。每篇都要求回扣前面三阶段，例如把 Attention 的访存模式与 cache / memory hierarchy 联系起来，把张量并行与分布式 shuffle 或 exchange 联系起来，把 graph optimization 与编译器 pass 联系起来。这样 AI 学习才会真正站在你已有系统背景之上。

四、每月主题安排

为了可执行，不建议同时铺太多主题。每月只抓一个主轴，最多两个相关子主题。1 月聚焦进程、线程、系统调用与上下文切换；2 月聚焦虚拟内存、页表、TLB、缺页异常；3 月聚焦文件系统、page cache、I/O 与 mmap；4 月聚焦锁、并发原语、调度与 NUMA；5 月聚焦哈希、树、堆、排序；6 月聚焦 bloom filter、skip list、LSM、图算法与动态规划；7 月聚焦 parser、AST、解释器与表达式求值；8 月聚焦 IR、CFG、SSA 与数据流分析；9 月聚焦优化 pass、LLVM 视角与数据库优化器映射；10 月聚焦 Transformer 基础与训练逻辑；11 月聚焦推理、KV cache、并行与显存；12 月聚焦 serving 系统、vLLM、部署与 AI infra 总结。

五、每周执行模板

为了避免计划看起来很美、执行起来散架，建议你固定周节奏。工作日每天安排 60 至 90 分钟，周末安排半天集中整理。具体来说，周一到周三用于阅读与理解主材料，周四用于实验、代码验证或工具观察，周五用于整理关键问题与笔记提纲，周末半天用于写一篇正式笔记或完成一项小练习。也就是说，平时做“输入 + 试验”，周末做“输出 + 收束”。对你这种已经在高强度工作中的工程师而言，这比每天高压猛冲更可持续。

一个典型周可以这样安排：周一读一章核心材料并列出问题；周二继续读并用自己的话复述；周三让 LLM 充当“反驳者”，针对你的理解提问；周四动手做实验，如 perf、mmap 小程序、表达式解析器、数据结构实现；周五把本周内容整理成一页要点；周末写成一篇 1000 至 2000 字的结构化笔记。这样一年下来，不只是“看了很多”，而是会形成一套你自己的知识仓库。

六、推荐资料

操作系统部分，优先读《Operating Systems: Three Easy Pieces》，配合 MIT 6.S081。计算机系统基础部分，读《CSAPP》中与内存、链接、虚拟内存、缓存相关章节。算法与数据结构部分，用《Algorithm Design Manual》或 Sedgewick 的《Algorithms》即可，不必贪多。编译器部分，先用《Crafting Interpreters》建立感性认识，再读《Engineering a Compiler》，之后配 LLVM IR 与 SSA 相关材料。AI 系统部分，前半段用高质量 Transformer 基础资料建立模型理解，后半段重点读 vLLM、DeepSpeed、TensorRT-LLM、Megatron-LM 等系统文章与文档。你的特点不是缺资料，而是资料过载，因此必须坚持“每阶段一主线，两三本核心材料即可”的原则。

七、LLM 的使用方式

LLM 在你的计划里，不是“老师”，而是“学习加速器”。最有效的用法有四种。第一种是概念澄清：看完一个章节后，让 LLM 用更紧凑的方式重述，并指出常见误区。第二种是反向追问：你先自己解释一个概念，再让 LLM 挑错、举反例、问边界条件。第三种是跨领域映射：让 LLM 把某个概念分别映射到操作系统、数据库、编译器、AI 系统中，帮助你建立统一视角。第四种是输出打磨：你先写笔记，再让 LLM 帮你检查逻辑漏洞、结构松散处与术语不准处。不要一开始就让 LLM 给你答案，那样很容易制造一种“读过总结就等于学会”的错觉。

你可以固定使用下面这四类提示词模板。其一：“请用严格的系统工程语言解释 X，分为定义、核心机制、性能影响、常见误区、与数据库/AI 系统的关系。”其二：“以下是我对 X 的理解，请指出其中错误、遗漏与不严谨之处，并给出反例。”其三：“请把 X 同时映射到操作系统、编译器、数据库、LLM serving 场景，比较它们的共性与差异。”其四：“下面是我的学习笔记，请帮我压缩语言、增强逻辑递进、改成书面表达。”这四类提示词足够你用一年，而且不会把思考权拱手让人。

八、输出机制

没有输出，学习几乎一定会松散。建议你把全年输出设计成三层。第一层是周记，每周一页，记录本周核心概念、疑问、实验与工作联系。第二层是月度笔记，每月一篇完整文章，围绕一个主题做结构化表达。第三层是季度复盘，每季度写一篇总结，回答三个问题：本季度真正掌握了什么，哪些地方只是“看过”，这些知识如何回到数据库内核或工作设计中。到年底时，你至少应当积累 40 份周记、12 篇月度文章、4 篇季度复盘，这本身就会构成一套相当可观的个人知识资产。

九、如何与工作结合

你最大的优势，是你不是纯学习者，而是有真实内核工作场景的工程师。因此每个主题都要主动寻找工作映射。学页表与虚拟内存时，思考 page cache、buffer pool、mmap scan 与 spill；学锁与调度时，思考 query concurrency、task scheduling、线程池与尾延迟；学哈希与树时，思考 join、group by、索引与字典结构；学 SSA 与优化 pass 时，思考 expression rewrite、MV 定义改写、规则推导与增量计算变换；学 Attention 与 KV cache 时，思考内存带宽、批处理调度、局部性与资源隔离。只有不断这样做，基础知识才会成为工作中的新工具，而不是停留在“业余学习项目”。

十、检验标准

到 2026 年底，你的检验标准不应是“读了多少书”，而应是三个能力是否显著增强。第一，解释能力：你是否能把一个概念讲清楚，而不是只会复述定义。第二，迁移能力：你是否能把一个领域的方法迁移到另一个领域，例如用编译器视角理解优化器，用操作系统视角理解 AI serving。第三，设计能力：你是否在工作中更容易识别问题本质，能提出更稳健的设计与优化判断。如果这三点提升明显，这一年就非常值。

十一、一个可直接照抄的每周计划

每周固定五步。第一步，选定本周唯一主题，例如“page cache”或“SSA”。第二步，读核心材料 2 至 3 次，每次都记录问题。第三步，做一个验证动作，可以是小程序、源码浏览、perf 观察或纸面推导。第四步，写出一版自己的解释，并让 LLM 充当审稿人。第五步，把它整理成正式笔记，并补一段“与数据库 / AI 系统的关系”。这个五步法简单，但极稳，适合长期执行。

十二、最后的建议

不要把这一年理解成“补课年”，而应理解成“认知重构年”。你已经具备很好的工程土壤，真正需要补的不是勤奋，而是结构。你的目标不是成为每门课都能考试满分的人，而是成为一个能把操作系统、编译器、算法、数据库、LLM/AI 系统放进同一张脑内地图的人。做到这一点，后续很多学习都会突然提速，因为你不再面对零散知识点，而是在不断扩展同一个系统世界观。

# Plan 2
## 一、核心学习原则（贯穿全年）

**"三角验证法"**：每个知识点必须经历三个维度才算真正掌握——
1. 能用自己的语言讲清楚（费曼技术）
2. 能写出代码验证（哪怕是玩具实现）
3. 能在工作中找到对应的影子（迁移锚点）

缺少任何一条，都只是"了解"，不是"掌握"。

## 二、上半年：计算机基础（1-6月）

### 第1-2月：操作系统

**核心目标**：理解"程序是如何跑起来的"

| 资源 | 用途 |
|---|---|
| 《Operating Systems: Three Easy Pieces》（OSTEP） | 主教材，免费在线，写法极好 |
| MIT 6.S081 | 配套 xv6 实验，写真实 OS 代码 |
| 《深入理解计算机系统》（CSAPP）第1、8、9章 | 补充进程/内存视角 |

**每周节奏**：
- 周一到周四：读 OSTEP 一章 + 做笔记
- 周五：用 Python/C 写一个对应的玩具实现（如：用 Python 模拟进程调度算法）
- 周末：对照你现在的工作，找到操作系统概念的投影（Query Engine 的线程池 = OS 的调度器）

**与你现有经验的连接点**：
```
OS 概念              →    Query Engine 对照
进程/线程调度         →    查询并发执行
虚拟内存/页表         →    Buffer Pool 管理
文件系统 inode        →    存储引擎页结构
系统调用开销          →    I/O 瓶颈分析
```

---

### 第3-4月：编译器原理

**核心目标**：理解"代码是如何变成执行的"，这对 Query Optimizer 是最直接的知识迁移

| 资源 | 用途 |
|---|---|
| 《Crafting Interpreters》（craftinginterpreters.com） | 免费全书，手把手实现一门语言，首选 |
| Stanford CS143 / Dragon Book 第1-4章 | 理论补充 |
| LLVM 官方 Tutorial（Kaleidoscope） | 接触真实工业编译器 |

**重点章节与 Query Optimizer 的映射**：
```
编译器阶段            →    Query Optimizer 对照
词法分析（Lexer）      →    SQL Parser Token 化
语法分析（Parser）     →    AST 构建
语义分析              →    类型检查 / 列解析
IR 生成               →    逻辑执行计划
代码优化              →    物理计划优化（谓词下推等）
代码生成              →    执行代码生成（Vectorized / JIT）
```

你会发现自己其实已经做了一半编译器的工作，只是没有系统化命名它。

### 第5-6月：算法与数据结构

**核心目标**：不是刷题，而是建立"看到问题自动映射到数据结构"的直觉

| 资源 | 用途 |
|---|---|
| 《算法》第4版（Sedgewick） | 最好的算法书之一，有 Java 代码 |
| LeetCode：按 Tag 刷，不按题号 | 每周 3-5 题，专注图论 + DP + 排序 |
| 《数据密集型应用系统设计》（DDIA）第3章 | 存储引擎数据结构，与你工作直接相关 |

**优先级排序**（对 Query Engine 工程师最有用的）：
1. B-Tree / B+Tree（存储引擎核心）
2. Hash Table / Bloom Filter（Join 实现）
3. 图算法（查询计划依赖图）
4. 动态规划（Join Order Optimization 的本质）
5. 堆 / 优先队列（Top-K 查询）

## 四、融会贯通的方法论

**"映射笔记"**：每学一个新概念，强制写一句话把它和已知概念连接：
> "LLM 的 KV Cache ≈ Query Engine 的 Hash Join Build 阶段的 Hash Table，都是用空间换时间"

**"反向教学"**：每月末，用 Rubber Duck 方式（对着橡皮鸭/ChatGPT/同事）讲一遍本月所学，讲不清楚的地方就是理解的漏洞。

**"工作中找影子"**：每周在日常工作里找一个概念，主动问自己"这背后的 OS/编译器/算法原理是什么"，记录下来。三个月后你会发现，工作本身就是最好的复习场。

**时间预算**：工作日每天 1-1.5 小时，周末 3-4 小时，全年约 500 小时，完全够用。关键是**不要断**，断了两周比没开始还难受。
