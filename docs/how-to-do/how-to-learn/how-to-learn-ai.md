面向 Query Engine 工程师的目标导向 AI 学习计划

以系统能力为锚点，完成从传统数据工程到 AI 时代工程师的能力迁移

在过去十余年里，软件工程师的核心竞争力，往往建立在几个相对稳定的维度之上：数据结构与算法、编程语言、系统设计、分布式架构、性能优化，以及对具体业务场景的抽象与落地能力。对于长期从事大数据基础设施、Query Engine、OLAP、分布式执行引擎、数据库内核相关工作的工程师而言，这些能力通常集中体现在执行器设计、查询优化、调度与资源管理、存储组织、向量化处理、缓存与索引、吞吐与延迟平衡等方面。

然而，随着大模型、RAG、Agent、推理系统和 AI 原生开发方式的兴起，工程领域正在发生一轮显著变化。变化并不意味着传统系统能力失效，恰恰相反，AI 时代对工程师提出了更高要求：不仅要理解模型与工具的能力边界，还要能够将复杂系统经验迁移到新的技术栈之中，并借此提升个人竞争力、工作效率与思考方式。

对于一名长期从事 Query Engine 或大数据基础设施的工程师而言，最重要的问题不是“是否要学习 AI”，而是“如何以最低的试错成本、最清晰的目标路径，将既有的系统能力迁移为 AI 时代的复利能力”。这篇文章尝试围绕这一问题，给出一套目标导向、可执行、兼顾技术深度与实践落地的学习规划。

⸻

一、为什么 Query Engine 工程师尤其需要系统性地学习 AI

许多工程师在面对 AI 相关话题时，容易走向两个极端。

一种极端是过度焦虑，误以为自己的传统系统背景会很快失去价值，进而试图从零开始学习模型训练、深度学习数学推导、各类流行框架，却因为路径过散、信息密度过高而很快陷入混乱。另一种极端则是轻视变化，将 AI 理解为“换了一个更强的搜索引擎”或“辅助写代码的工具”，从而忽视其对软件开发范式、知识工作流程、产品形态和基础设施架构的深刻影响。

从实际趋势来看，这两种判断都不准确。

AI 的快速发展，确实正在重塑软件工程的若干环节，例如代码生成、文档处理、知识检索、问答系统、工作流自动化、数据分析、异常检测、技术调研、产品交互方式等。但与此同时，真正能够构建高质量 AI 系统的人，往往并不是仅会调用几个 API 的应用开发者，而是那些兼具系统能力、数据组织能力、性能意识、架构抽象能力与工程落地能力的工程师。

而这恰恰是 Query Engine 工程师的强项。

从系统视角看，很多 AI 应用与基础设施问题，与数据库和执行引擎问题有着非常强的结构相似性。例如：
	•	RAG 的核心并非“调用一个向量库”，而是检索、过滤、排序、上下文构造、缓存、增量更新、质量评估与成本控制的整体系统设计；
	•	LLM 推理服务涉及 KV Cache、批处理、调度、公平性、资源隔离、吞吐与延迟平衡，这些都与数据库系统中的执行调度和资源管理高度相通；
	•	Agent 的本质并不是“会思考的神秘体”，而是模型、工具、状态管理、规划流程与反馈机制的组合；
	•	AI 原生开发强调将模糊需求拆解成结构化工作流，这与高级工程师对复杂系统进行模块化、流程化、抽象化的能力直接相关。

因此，对于 Query Engine 工程师而言，学习 AI 并不是另起炉灶，而是一次能力迁移：将原本在数据系统、执行引擎和复杂工程中的积累，转化为 AI 时代更具价值的跨域能力。这种迁移一旦完成，带来的不仅是新技术理解能力，更是新的职业叙事与更强的长期竞争力。

二、AI 时代，Query Engine 工程师真正需要补足的能力是什么

在开始制定学习计划之前，必须先明确目标。如果目标不清晰，学习很容易沦为工具收藏、文章堆积与概念复述。对于有多年系统研发经验的工程师而言，学习 AI 的重点并不是“尽可能多地知道一切”，而是围绕以下四类核心能力展开。

1. 将 AI 作为工程生产力工具的能力

首先需要具备的是“会用”的能力，也就是将 AI 从偶尔尝试的新工具，转变为日常工作中的稳定外部助理。这种能力看似浅层，实际却是最容易带来立竿见影收益的一层。

工程师可以将 AI 用于以下场景：
	•	阅读与总结复杂技术文档、论文、Release Notes、设计文档；
	•	快速梳理代码模块职责与调用链；
	•	生成初版设计文档、PR 描述、测试用例、周报与调研摘要；
	•	协助进行日志分析、异常排查、思路发散与技术方案对比；
	•	在多语言栈之间完成代码片段转换与重构建议；
	•	作为面试准备、知识整理与学习复盘的辅助工具。

这一层并不是“让 AI 替代工程师判断”，而是将 AI 用作高效的草稿机、问题拆解器、结构整理器和思维陪练者，从而释放时间与精力，使工程师把更多注意力放在关键决策和复杂问题上。

2. 理解大模型、RAG 与 Agent 的基本原理

第二层是“会懂”。如果不理解 AI 系统的基本工作方式，工程师就很容易被市场话术、框架名词和产品包装带偏，形成模糊而不可靠的判断。

这一层需要掌握的，并不一定是深度学习中最底层的数学推导，而是足以形成工程判断的概念框架，例如：
	•	Transformer、Attention 的作用与基本机制；
	•	Pretraining、SFT、RLHF、Reasoning Model 等阶段的大致区别；
	•	Token、Context Window、KV Cache 等关键概念；
	•	推理过程中的主要瓶颈：算力、显存、带宽、延迟、吞吐；
	•	Embedding 与向量检索的本质；
	•	RAG 的完整链路，包括检索、排序、上下文拼接、答案生成与评测；
	•	Agent 的组成：模型、工具、记忆、计划、反馈回路；
	•	为什么评测比 Prompt 更重要，以及如何理解稳定性、可解释性与幻觉问题。

对于 Query Engine 工程师而言，这部分内容尤其适合用“系统工程视角”理解，而不是停留在概念背诵层面。换句话说，在学习时应持续追问：输入输出是什么、状态在哪里、瓶颈在哪里、哪些部分可缓存、哪些部分适合预计算、哪些决策会带来质量与成本之间的权衡。

3. 设计并实现一个完整 AI 系统的能力

第三层是“会做”。理解概念与使用工具只是开始，真正的能力要通过完整项目的搭建来建立。只有亲手实现一个最小但闭环的 AI 系统，才会真正遇到并理解许多关键问题：文档如何切分，元数据如何设计，检索如何排序，引用如何生成，评测如何落地，质量如何度量，延迟如何优化，工作流如何抽象。

对于 Query Engine 工程师而言，这类实践项目并不需要追求华丽界面或大而全功能，而应强调“完整性”和“工程真实感”。例如：
	•	面向数据库技术资料的 RAG 助手；
	•	面向 GitHub 仓库的源码理解与问答助手；
	•	面向个人技术研究的知识库与总结系统；
	•	面向调试、Benchmark、性能分析的工作流自动化工具。

这些项目会帮助工程师真正形成对 AI 工程链路的理解，而不仅仅是停留在工具拼接层面。

4. 理解 AI 对职业路径与软件形态的影响

第四层是“会想”。技术学习最终不能只停留在“掌握了某个工具”，而应该进一步理解：AI 将如何改变软件产品、工程工作流、组织分工、人才结构以及个人职业路径。

对于资深工程师而言，尤其需要清楚以下判断：
	•	哪些能力会在 AI 时代升值，例如系统设计、性能优化、数据组织、复杂流程自动化与跨域抽象能力；
	•	哪些能力会相对贬值，例如低复杂度 CRUD、机械式文档加工、缺乏判断力的简单胶水代码开发；
	•	自己当前的背景如何与 AI 产生结合点，能够进入哪些新的角色，如 AI Infra Engineer、RAG Engineer、AI 原生数据平台工程师、智能研发工具链工程师等；
	•	未来三到五年，自己的学习与输出应围绕哪些主题持续累积，才能形成真正的长期优势。

这一层决定了工程师是否只是“跟上了工具”，还是开始真正建立起 AI 时代的个人技术方向。

⸻

三、Query Engine 工程师最值得重点关注的 AI 方向

由于时间和精力有限，学习 AI 必须强调方向选择。平均用力通常意味着平均平庸。结合 Query Engine 工程师的知识结构，以下几个方向最值得重点投入。

1. LLM 推理系统与 AI 基础设施

这是与数据库执行引擎、资源管理、调度系统最接近的方向之一。相关主题包括：
	•	Continuous Batching；
	•	Paged Attention；
	•	KV Cache 管理与复用；
	•	Prefix Cache；
	•	Speculative Decoding；
	•	多租户调度与 Admission Control；
	•	推理延迟与吞吐优化；
	•	长上下文支持与资源管理；
	•	CPU / GPU / 显存 / 网络的瓶颈分析。

这一方向对于有执行引擎、调度器、资源隔离、队列管理经验的工程师非常友好，因为很多问题本质上仍是系统问题，只不过处理对象从查询计划变成了 Token 生成过程。

2. RAG 与检索增强系统

这一方向与数据库人的天然契合度极高，因为其核心并不是“把文本塞给模型”，而是一个完整的数据处理与查询系统。典型主题包括：
	•	文档清洗与结构化解析；
	•	Chunk 切分策略与语义边界保持；
	•	Embedding 建模与索引构建；
	•	Hybrid Search：倒排检索与向量检索的组合；
	•	Rerank 与多阶段检索；
	•	元数据过滤、权限控制、时效性管理；
	•	引用与证据链构造；
	•	质量评测、延迟优化与缓存设计；
	•	增量更新与在线索引维护。

对于 Query Engine 工程师而言，很多熟悉的问题会再次出现：该预计算什么、该实时计算什么、如何进行查询重写、如何做多阶段执行、如何平衡效果与成本、如何建立评测基准。

3. AI for Engineering：用 AI 反哺研发工作流

这是最容易直接提升个人工作效率的方向，也最适合工程师在日常工作中快速建立正反馈。常见场景包括：
	•	结合日志、指标、Profile、Flame Graph 进行异常原因候选生成；
	•	阅读大型代码库、PR、Issue、设计文档并自动形成模块说明；
	•	生成 Benchmark 总结、测试用例扩展、代码评审清单；
	•	协助进行技术调研、论文摘要、版本演进对比与知识库构建；
	•	为数据库优化器、执行器、存储引擎相关工作提供辅助解释与方案发散。

这个方向不一定最具“话题性”，但往往最有实际价值，因为它直接作用于工程师每天都在处理的工作流。

4. Agent 与工作流自动化

Agent 并非某种神秘智能体，而是让模型能够在工作流中调用工具、读写状态、进行任务分解和反馈迭代的一种系统范式。对于工程师而言，其价值主要体现在复杂知识工作流程的半自动化，例如：
	•	读取 Issue，查找相关代码与 PR，生成问题分析与初步修复建议；
	•	解析 Release Notes，提取 Feature、性能优化点与趋势变化；
	•	结合监控、日志与配置，生成故障排查候选路径；
	•	阅读设计文档并输出评审清单、风险点与待确认问题；
	•	构建面向技术研究或团队交接的自动化知识整理系统。

掌握 Agent 和 Workflow 的价值，在于工程师可以逐步将自身脑力劳动中的重复部分抽象出来，形成半自动化的辅助系统。

四、一个目标导向的 AI 学习总原则

在正式展开计划之前，还需要先明确几条总原则。没有这些原则，计划很容易在执行中变形。

第一，以能力迁移为主，而非身份重置

对于有多年 Query Engine 经验的工程师而言，不应以“从零转行做算法工程师”为目标，而应以“将已有系统能力迁移到 AI 场景”作为学习主轴。这样的路径更符合优势积累，也更容易形成差异化竞争力。

第二，以项目闭环为核心，而非知识堆积

学习 AI 时，最容易出现的问题是看了大量文章、收藏了大量工具、记住了大量术语，但始终缺乏一个真正完成过的系统。因此，必须坚持“边学边做”，并且至少完成两个小而完整的实践项目。项目的完整度比花哨程度更重要。

第三，以工作流嵌入为目标，而非 Demo 展示

学习的终点不是做一个公开演示用的小工具，而是将 AI 能力嵌入日常工作流，真正改变自己的做事方式。能够提升文档阅读、调研效率、问题拆解、代码理解与输出质量，才说明学习有真实价值。

第四，以评测和复盘为核心，而非依赖感觉

AI 项目与传统系统一样，需要评测、反馈和迭代。很多工程师在使用 AI 工具时容易“凭感觉认为效果不错”，但没有可靠验证。应当尽早建立对效果、质量、准确性、引用可信度和延迟成本的基础评估习惯。

第五，以长期复利为导向，而非短期追逐热点

AI 领域变化极快，框架、模型、产品层出不穷。如果把精力过多放在追逐最热概念上，很容易陷入表层忙碌。真正值得长期投入的是那些底层稳定、可迁移、可复用的能力，如系统抽象、检索与推理链路理解、性能优化、工作流自动化与知识组织能力。

五、面向 Query Engine 工程师的三个月目标导向 AI 学习计划

下面给出一套为期三个月的学习计划。这一计划并不追求覆盖所有 AI 领域，而是强调路径清晰、目标明确、阶段产出可验证。

⸻

第一阶段：建立 AI 基础地图（第 1 至 4 周）

阶段目标

这一阶段的核心目标不是立即写出复杂系统，而是尽快形成完整的知识框架，明确 AI 相关技术的主要组成、关键问题与工程边界，建立对大模型、RAG、Agent 和推理系统的结构化理解。

学习重点

在这一阶段，应当重点理解以下内容：
	1.	大模型的基本结构与训练流程
了解 Transformer、Attention、Token、Embedding、上下文窗口、推理过程等基础概念，以及预训练、监督微调、偏好对齐、推理增强等环节的大致作用。
	2.	推理系统的核心问题
理解推理过程中的主要系统瓶颈，例如显存限制、带宽瓶颈、KV Cache 管理、Batching、延迟与吞吐的平衡、多租户场景下的调度问题。
	3.	RAG 的完整链路
理解文档处理、Chunk 切分、Embedding 建模、索引构建、检索、重排序、上下文构造、答案生成和效果评测的整体流程。
	4.	Agent 与工作流自动化
理解 Agent 的基本模式：模型调用工具、维护状态、执行计划、进行反馈迭代，掌握其与传统自动化脚本的区别与联系。
	5.	AI 评测的基本意识
理解为什么“能跑起来”不等于“可用”，以及为什么在 AI 项目中，评测、引用可信度、稳定性和错误分析极其重要。

建议产出

这一阶段结束时，建议至少完成以下输出：
	•	一份“AI 基础概念地图”，覆盖大模型、RAG、Agent、推理系统四个板块；
	•	一份“LLM 推理系统关键瓶颈表”，从系统视角整理 CPU、GPU、内存、显存、网络、调度、缓存等问题；
	•	一份“RAG 系统链路拆解图”，标注每个阶段的输入、输出、关键设计点和可能瓶颈；
	•	一份“Agent 工作流抽象笔记”，总结模型、工具、记忆、计划和反馈的关系。

阶段要求

这一阶段不要急于追求深度学习数学细节，也不要急于使用过多框架。重点应是形成整体地图，建立系统级理解。同时，在学习过程中，始终应用数据库人的思维方式去分析 AI 系统：输入输出、状态管理、预计算、缓存、代价权衡、吞吐与延迟平衡等。

⸻

第二阶段：搭建第一个完整 AI 项目——数据库技术资料 RAG 助手（第 5 至 8 周）

阶段目标

这一阶段的核心目标是亲手搭建一个完整的 RAG 系统，形成对 AI 应用基本架构的工程理解。项目应尽量选取自己熟悉、能够判断质量的领域，以降低无效试错成本。对 Query Engine 工程师而言，最适合的题目是“数据库技术资料 RAG 助手”。

项目目标定义

输入数据可以包括：
	•	数据库产品文档；
	•	Release Notes / Changelog；
	•	官方博客；
	•	论文 PDF；
	•	GitHub PR、Issue、设计文档；
	•	自己整理的读书笔记或调研材料。

系统应至少具备以下能力：
	•	支持文档导入与索引；
	•	支持基于问题的检索与问答；
	•	支持返回引用来源；
	•	支持按主题进行总结；
	•	支持对多个系统进行对比性问答。

在项目中重点学习的问题

这个项目真正的价值，不在于“做出一个能聊天的页面”，而在于在实现过程中思考以下关键问题：
	1.	文档如何切分
不同类型文档（规范文档、博客、PR、论文）是否应采用不同的 Chunk 策略？如何避免语义断裂？如何保留标题层级和上下文信息？
	2.	元数据如何组织
文档类型、来源、作者、时间、产品名称、主题标签、版本信息等元数据是否应保留？它们如何帮助后续检索和过滤？
	3.	检索链路如何设计
是否只做向量检索，还是结合关键词检索与重排序？是否需要多阶段检索？如何处理术语和代码片段？
	4.	引用与可信度如何处理
如何将最终回答与具体证据片段绑定？如何降低模型胡编的概率？如何让总结更接近原文而非“凭印象生成”？
	5.	评测如何开展
可以自拟一批真实问题，例如“某数据库在近两年中引入了哪些重要执行引擎优化”“某个版本的主要变化是什么”“某项特性在哪个 PR 中首次出现”，用来测试系统的回答质量和引用准确率。

建议产出

这一阶段结束时，应至少完成：
	•	一个可运行的最小 RAG 系统；
	•	一份项目架构说明文档，解释数据流转、索引构建、检索、生成和评测方式；
	•	一份“问题样例与评测结果”文档；
	•	一份项目复盘，记录在 Chunk、检索、引用、Prompt 和质量方面的主要问题与下一步优化方向。

阶段意义

完成这个项目后，工程师通常会对 AI 应用系统有一次明显“落地式理解”。很多此前停留在文章中的概念，会在实践中变得具体而真实。例如，检索效果不稳定并不只是模型问题，也可能是 Chunk 粒度过粗、元数据缺失或检索策略单一所致；引用不准确不只是 Prompt 问题，也可能是上下文构造方式有缺陷。

⸻

第三阶段：搭建第二个项目——代码库理解与工作流辅助工具（第 9 至 12 周）

阶段目标

如果第一个项目主要帮助建立对“文档型 AI 系统”的理解，那么第二个项目的重点应转向工程场景，使 AI 学习成果与日常研发工作更紧密结合。最合适的方向是“代码库理解与工作流辅助工具”。

项目目标定义

项目输入可以包括：
	•	某个开源数据库或自己熟悉项目的源码；
	•	PR、Issue、Commit Message；
	•	设计文档、测试文件、注释和构建脚本。

系统目标可以包括：
	•	回答某个模块或类的职责；
	•	给出与某个问题相关的代码文件与调用链线索；
	•	帮助理解某个功能的演进过程；
	•	根据 Issue 生成相关代码阅读路径；
	•	从 PR 与文档中提取变更背景与设计动机。

这一阶段重点学习的问题
	1.	如何为代码建立可检索表示
代码与普通文档不同，它具有函数、类、模块、调用关系、文件路径等结构特征。如何在索引时保留这些结构信息，是这一阶段的重要思考点。
	2.	如何结合多源信息
真正的工程问题往往不能只靠代码回答，还需要结合 PR、Issue、文档、测试用例和注释。如何融合这些信息，是构建高质量工程辅助工具的关键。
	3.	如何设计更接近实际工作的问答方式
与其问“这个函数做什么”，不如用更真实的场景问题，例如“某个运行时过滤优化在执行链路中由哪些模块配合完成”“为什么某次性能回退可能与某个调度策略有关”。
	4.	如何辅助问题拆解，而不是替代思考
该类工具的价值通常不在于直接给出最终结论，而在于帮助工程师快速缩小问题空间、形成阅读路径和候选假设。

建议产出

这一阶段结束时，建议完成：
	•	一个最小可运行的代码库理解工具；
	•	一套真实工程问题样例；
	•	一份使用复盘，说明工具在日常阅读代码、设计排查路径和知识交接中能提供哪些帮助；
	•	一份“工程知识型 AI 系统设计笔记”，总结文档型与代码型系统在索引、检索、上下文构造上的差异。

⸻

六、学习完成后，如何把 AI 真正嵌入日常工作流

三个月的学习与项目实践只是起点。真正能够体现价值的，是将 AI 从“额外实验”嵌入到日常工作中，逐渐形成新的工作习惯与思考方式。对于 Query Engine 工程师而言，可以重点从以下几个方面推进。

1. 将 AI 用于技术阅读的结构化提炼

面对论文、Release Notes、技术博客或设计文档时，可以先让 AI 帮助完成结构抽取，再由自己进行批判性复核。阅读时建议固定输出模板，例如：
	•	该文章或版本解决了什么问题；
	•	核心设计点是什么；
	•	与旧方案相比有哪些变化；
	•	主要代价、边界与局限是什么；
	•	对自身系统或工作有何启发。

长期坚持后，不仅阅读效率会提升，思考的结构化程度也会显著增强。

2. 将 AI 用于代码理解与问题拆解

面对复杂代码库、异常日志、性能问题或设计改动时，可以先让 AI 辅助形成问题树、候选原因和验证路径，再由自己进行实验与判断。这种方式能够显著降低“从零开始定位问题”的认知开销。

3. 将 AI 用于文档与知识资产沉淀

设计文档、交接文档、PR 描述、性能分析报告、Benchmark 总结、会议纪要等内容，都可以借助 AI 生成初稿或完成结构梳理，然后再由工程师补充关键事实和判断。这样做能够减少机械整理时间，同时提高知识沉淀的频率。

4. 将 AI 用于工作流自动化

许多重复性的研发活动，其实都可以逐步半自动化，例如：
	•	汇总版本变更并生成简报；
	•	从 Issue 和 PR 中抽取交接清单；
	•	解析日志与监控并生成候选排查路径；
	•	生成代码评审检查项；
	•	构建个人技术研究资料库。

真正高效的工程师，并不是比别人更擅长忍受重复劳动，而是更早学会将重复劳动系统化、自动化。

5. 保持“AI 是假设生成器，而非真相来源”的态度

这是最重要的一点。AI 可以极大提升思考速度与发散能力，但它并不天然等于正确。尤其在技术判断、架构设计、性能分析和故障定位中，工程师必须始终保留人工复核与证据意识。高级工程师在 AI 时代最值钱的能力，不是生成内容，而是对生成结果进行验证、筛选、纠错和提纯。

⸻

七、AI 时代，Query Engine 工程师的长期竞争力应如何重塑

经过一段时间的学习与实践后，工程师需要逐步完成自身职业定位的升级。过去可以将自己定义为“数据库内核工程师”“执行引擎工程师”“大数据基础设施工程师”，而在 AI 时代，更值得追求的目标，是成为“理解数据系统、推理系统与工作流自动化的 AI 原生系统工程师”。

这一升级意味着：
	•	不再只是优化传统查询执行路径，而能够理解和优化推理服务、检索链路与智能工作流；
	•	不再只是实现数据库内部模块，而能够将数据组织、检索、缓存、调度和评测能力迁移到 AI 系统中；
	•	不再只是依赖人工完成知识工作流程，而能够设计半自动化、可复用、可评测的工程辅助系统；
	•	不再只是掌握某一门技术，而是具备跨越数据库系统、AI 应用、工作流自动化和产品抽象之间边界的能力。

未来几年，这类工程师通常会处于一个很有价值的位置：既不像纯模型研究者那样高度依赖算法背景，也不像传统基础设施工程师那样容易被新范式边缘化，而是能够站在系统、数据与 AI 的交叉点上，构建真正有工程价值的新一代工具与平台。

⸻

八、结语：学习 AI，不是追逐神话，而是为自己增加新的杠杆

对于长期从事 Query Engine、大数据基础设施和数据库系统研发的工程师而言，AI 并不是一个需要仰视的神秘领域，也不是一阵可以轻视的技术风潮。它更像是一组正在重写软件生产方式、知识工作方式和系统交互方式的新工具、新架构与新范式。

真正值得做的，不是焦虑地追逐所有热点，也不是把自己已有的系统经验全部推翻，而是在清晰目标的牵引下，完成一次有方向、有方法、有产出的能力迁移。这样的迁移，既能帮助工程师提升工作效率与思考方式，也能在未来的技术竞争中形成新的位置感与长期复利。

从这个意义上说，学习 AI 的关键并不在于“知道多少新名词”，而在于是否真正做到了以下几点：能够更快地获取和组织信息，能够更结构化地分析问题，能够亲手完成一个小而完整的 AI 系统，并能够将这些能力持续嵌入自己的实际工作中。

这条路并不需要冒进，也不需要盲目。对有深厚系统背景的工程师而言，最优路径往往不是推倒重来，而是把原本擅长的东西，接到新的技术杠杆上。这样一来，AI 就不再只是外部变化，而会逐渐成为个人能力体系中的一部分。到了那时，工程师面对这轮变化时，不会只是被动适应，而是真正具备了借势而行的能力。
