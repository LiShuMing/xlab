MV整体工作，在我理解起来核心是在于“复用”；其核心价值在于最大化利用内核引擎的优势，去解决上层业务的“重复性”的问题。所以这也决定了MV工作的一些性质：
- 距离业务比较近，使用场景、使用方式比较多，使用技巧较多，暴露的参数多，用好需要些心智；
- 同内核引擎打交道的地方多，有些地方使用的不对，或者不符合预期，容易暴露性能问题；

## MV工作分类
同步MV
- 核心逻辑：
  - 类似于表的Index构建（原来就是RollupIndex)，复用了SchemaChanging（刷新存量历史数据） + TabletSink实时导入的能力（新增数据），保障MV同基表在一个事务中提交，实时可见；
- 目前基本处于维护状态，但不时仍然有用户在用(kaspi），发现了几个bug；
- 功能：
  - 单表：projection或者聚合；
  - 支持了通用聚合函数，支持了Where表达式、Limit表达式；
  - 支持存算一体、存算分离；
- 局限：
  - 数据分布(分区、分桶）、生命周期跟基表绑定；
  - 无法支持多表复杂Query场景。

异步MV
- 核心逻辑：
  - PCT: 基于分区变化的分区级别增量物化视图构建；
  - IVM: 基于Delta变化的增量物化视图构建，目前IVM仍处于MVP 实验阶段，整体框架层面不管是建表约束、透明加速或者同PCT兼容，还是沿用PCT的整体逻辑，当IVM成熟以后，理论上可以重新思考拓展现有约束（比如是否一定要维护基表到MV的Mapping关系），去拓展MV的场景、边界；
- MV 创建
  - 目前还是沿用PCT支持边界进行编码的，一个MV的创建必须跟基表存在分区映射（1:1、上卷、下钻），但这些映射关系在FE测必须是可推导的，否则就没法支持。
  - MV 支持Range的基表分区类型，List的基表分区类型；但不能支持既有Range、List的分区基表类型的分区MV。
  - MV支持多事实表分区级别增量刷新，eg:多个事实表join/union，同时按照天进行增量调度刷新；
- MV 改写
  - 时效性补偿
  - 基于SPJG的改写
  - 支持基于UK/FK Constraint裁剪冗余MV Tables
  - 支持View Based MV Rewrite
  - 基于文本的改写
  - 支持聚合下推的改写
  - 支持时序场景的改写
- MV 刷新
  - 时效性补偿
  - 基于分区的PCT刷新
    - 支持各种数据源
  - 基于IVM的增量刷新
    - Iceberg
    - Paimon
  - 默认刷新策略：
    - 默认开启了Spill；
  - 资源隔离
    - Resource Group
    - Warehouse
- MV调度能力
  - 复用Task的调度管理能力
    - Task/TaskRun
      - Task: MV
      - TaskRun: MV的一次刷新的最小单元，MV一次刷新可能会刷新多个分区，根据partition_refresh_number (默认为1）拆分成多个TaskRun，每个TaskRun刷新partition_refresh_number个分区；
    - 底层基于ScheduledExecutorService.scheduleAtFixedRate 按照定义周期触发执行TaskRun调度；
    - 所有的PendingTaskRuns 维护在一个全局队列中（TaskRunFIFOQueue)，目前的策略是FIFO，保障全局刷新的公平调度；
  - 调度方式：
    - MANUAL
    - ASYNC：事件式（目前只能感知基表为NativeTable）的异步调度；
    - PERIODIC: 周期调度
- MV的可观测性
  - MV刷新
    - information_schema.task_runs
    - information_schema.materializedd_views
    - Profile中增加了不少MV的可观测性指标
    - SHOW MATERIALIZED VIEWS
    - MV Metrics
  - MV改写
    - trace mv logs <query>
    - Profile中增加了不少MV的可观测性指标
    - MV Metrics
- 外围的一些功能
  - MV与SchemaChange
  - MVActiveChecker
    - 后台任务，默认开启，自动检测非手动inactive 的MV，自动尝试active 该MV使其可用（有些因为基表recreate或者schema change导致的）；
  - MV的的测试
    - SQL-Tester
    - StarRocksTest

## MV思考&改进
  - MV支持边界
    - 目前创建MV依然需要在其定义Query中将基表的分区列作为最终输出列才行，否则没法创建MV；
      - 是否可以简化此操作，如果有选项保障基表->MV是意义映射的话？
    - OlapTable希望能统一Range/List 分区类型，MV理论上也可以支持通用的分区映射；
    - 如果IVM的增量刷新能够成熟以后，就可以在创建MV的时候强制要求基表和MV之间的映射关系了（eg: 基表是非分区，MV可以是分区表），这时候就要重新思考当前的MV约束了，同时也有其他的一些问题（可以参考下BigQuery的MV和Snowflake Dynamic Table)：
      - 如何再进行Query Rewrite？
      - 如何在PCT和IVM之间进行切换？
    - 多事实表分区级别增量刷新；
    - MV同基表的生命周期解耦问题
        - 目前这块遗漏问题较多，暂未形成统一的产品设计。
  - MV改写增强
    - 改写局限
      - 改写完备性，支持非SPJG补偿: Grouping Set/Set Operator/Window/Limit/TopN/Union等
      - 基于Text文本的改写能力比较局限（相较于Oracle)，还可以支持更多的Pattern，比如部分Output不对齐场景；
      - View Rewrite不支持Union补偿，涉及到基表分区变化的场景，依然退到SPJG改写补偿；
      - MV改写规则跟现有RBO存在冲突，目前基于多个Stage进行改写能够规避部分改写rule冲突，但也增加了耗时；
      - 改写目前还依赖CBO阶段的改写，也可能导致改写耗时；
    - 改写性能问题
      - 抽象MVCache 减少重复计算，将每次QueryRewrite涉及MV的均需要做的一些action做更好的抽象；
        - 目前MV Cache这块有些分散，有好几处散落在Query改写、MV刷新的Cache，可以考虑维护MV级别更统一的Cache减少所有的重复计算；
        - 现在把MV用于改写的OptExpression，因为有时用户的Query依赖大量外部表，该load操作可能会比较耗时；目前该Load操作是不阻塞Query的，在其Load完之后会放置到了CachingMvPlanContextBuilder Cache中，其更新时机：
          - FE重启的时候；
          - 第一次访问；
          - 每次MV Inactive/Active；
          - 调整enable_query_rewrite；
      - 目前每次Query查询，都需要检测下其与关联MV的时效性，这块有很多重复计算及性能问题：
        - 因为每次时效性检测，都需要遍历MV和基表的所有分区，获取PartitionDiff；虽然能够保障一致性，但在分区较多或者有外表场景，该操作可能会有很多性能问题；
        - 之前有个思路，能否在刷新的时候维护部分改写的补偿逻辑，force_mv  做了部分尝试，但有机会还是思考下，如何较少Query改写时的时效性检测的性能问题。
      - MV改写有些 场景下耗时较重，能否同PlanCache结合减少FE测耗时：
        - 保存相同(Pattern)的Query改写后的Plan，减少parser/analyzer/mv rewrite的代价；但时效性该如何保障？
  - MV刷新能力增强：
    - 刷新性能问题
      - 类似于MV改写，抽象MVCache，减少不必要重复计算（目前已经添加了部分Cache到CachingMvPlanContextBuilder），但可以更加统一；
    - 明确Rolling Refresh场景（比如用户希望只刷新最近7天的数据）；
    - 外部数据源能力增强
      - JDBC 
      - Hudi
  - MV调度增强
    - 支持Crontab通用调度能力；
    - 支持高频调度能力，不能再简单地基于JDK内置的ScheduledExecutorService；
    - 支持更丰富的调度策略；
      - 支持更丰富的DAG调度（级联刷新），思考如何替代外部的调度平台场景（airflow/dophine scheduler）；
      - 支持重试；
      - 支持优先级；
      - 支持更好的血缘调度，在事件调度的语义下，如何更好地支持更多的基表数据源；
  - MV运维能力增强
    - MV改写诊断，输出更多可阅读、可解释的结构化信息，方便运维DBA或者用户直接获取改写失败原因；
    - 结合BYOC产品，输出针对MV更好的可观测性：每个MV的刷新进度（时效性），命中次数，收益等；
  - MV代码维护
    - 不管是MV Create/MV Query/MV Refresh，代码经过很多人手，命名或者抽象都不统一，在4.x的时候做过部分的统一，但依然还有很多历史Code不好动；有机会可以进行更彻底的抽象：
      - MV or Mv；
      - AMV or MTMV
      - PCT
      - Ivm or Tvr
  - MV测试
    - 在StarRocksTest已经增加了大数据量(1W+个MV)的基本测试（测试对FE重启的影响），同时也有一个MVTestFramework，包含MV Lifecycle的一些基础Case，但MV的使用场景太广泛，建议有机会可以再基于该Framework再添加多一些的测试Case。
    - 目前MV一直缺少在StarRocksBenchmark的测试，建议可以在StarRocksBenchmark添加一些Corver Case:
      - 基表（外表、内表）有很多分区，刷新、改写Performance（同不该写对比）；
      - MV有很多基表（10/50/100），刷新、改写Performance（同不该写对比）；
      - 添加SSB/TPCH/TPCDS 100GB Scale的例行化Benchmark，观察版本release的变化情况；
