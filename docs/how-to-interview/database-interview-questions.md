# 数据库面试题精编

> 本文整理了 MySQL、Redis 等数据库相关的面试高频问题，涵盖索引、事务、锁、日志、性能优化等核心知识点。

---

## 一、MySQL 基础

### 1.1 存储引擎对比

| 特性 | InnoDB | MyISAM | Memory |
|------|--------|--------|--------|
| **事务支持** | 支持 | 不支持 | 不支持 |
| **行级锁** | 支持 | 不支持（表锁）| 表锁 |
| **外键** | 支持 | 不支持 | 不支持 |
| **崩溃恢复** | 支持（redo log）| 不支持 | 不支持 |
| **MVCC** | 支持 | 不支持 | 不支持 |
| **全文索引** | 5.6+支持 | 支持 | 不支持 |
| **适用场景** | 高并发、事务 | 读多写少、日志 | 临时表、缓存 |

**为什么默认使用 InnoDB？**
- 支持事务，保证数据一致性
- 支持行级锁，并发性能高
- 支持崩溃恢复

### 1.2 索引原理

#### B+ 树结构

```json
                    [10, 20, 30]
                   /     |     \
            [5,8]    [12,15,18]    [25,28]    [35,40]
           /   \      /    |    \    /    \      /    \
          1→5  8→9  10→11 12→14 15→17 18→19 20→24 25→27 ...
```

**B+ 树特点**：
- 非叶子节点只存储键值，不存储数据
- 所有数据存储在叶子节点
- 叶子节点通过指针相连（便于范围查询）
- 树高度低，IO 次数少

**为什么用 B+ 树不用红黑树？**
- B+ 树高度更低（多路），磁盘 IO 更少
- 红黑树是二叉树，大数据量时高度太高
- B+ 树支持顺序访问（叶子节点链表）

**为什么不用 B 树？**
- B 树非叶子节点也存数据，导致每个节点存储的键值更少
- B+ 树查询性能更稳定（都在叶子节点）
- B+ 树范围查询效率更高（叶子节点链表）

#### 索引类型

| 索引类型 | 说明 | 特点 |
|---------|------|------|
| **主键索引** | 主键上的索引 | 唯一，非空，聚簇索引 |
| **唯一索引** | 唯一约束的索引 | 允许 NULL |
| **普通索引** | 普通列上的索引 | 无特殊约束 |
| **全文索引** | 文本搜索 | 类似搜索引擎 |
| **覆盖索引** | 查询字段都在索引中 | 无需回表 |

#### 聚簇索引 vs 非聚簇索引

| 特性 | 聚簇索引 | 非聚簇索引（二级索引）|
|------|---------|-------------------|
| **叶子节点** | 存储完整数据行 | 存储主键值 |
| **数量** | 一个表只有一个 | 可以有多个 |
| **查询** | 直接返回数据 | 可能需要回表 |
| **插入** | 影响数据物理存储顺序 | 不影响 |

**回表查询**：
```sql
-- 假设 name 是二级索引，查询其他字段需要回表
SELECT * FROM user WHERE name = '张三';
-- 1. 在 name 索引找到主键 id
-- 2. 通过 id 在聚簇索引查找完整数据
```

**覆盖索引避免回表**：
```sql
-- name 和 age 构成联合索引
SELECT id, name, age FROM user WHERE name = '张三';
-- 索引中包含所有查询字段，无需回表
```

### 1.3 索引设计原则

**应该建索引的列**：
- 经常作为查询条件的列（WHERE）
- 经常用于连接的列（JOIN）
- 经常用于排序的列（ORDER BY）
- 经常用于分组的列（GROUP BY）
- 区分度高的列

**不适合建索引的列**：
- 区分度低的列（如性别）
- 频繁更新的列（维护索引开销）
- 很少查询的列
- 小表（全表扫描更快）

**联合索引最左前缀原则**：
```sql
-- 联合索引 (a, b, c)
WHERE a = 1              -- 使用索引
WHERE a = 1 AND b = 2    -- 使用索引
WHERE a = 1 AND b = 2 AND c = 3  -- 使用索引
WHERE b = 2              -- 不使用索引（缺少 a）
WHERE a = 1 AND c = 3    -- 只使用 a（b 缺失）
WHERE a LIKE '%x'        -- 不使用索引（最左前缀模糊）
```

## 二、事务与并发控制

### 2.1 ACID 特性

| 特性 | 说明 | 实现机制 |
|------|------|---------|
| **原子性 (A)** | 事务要么全成功，要么全失败 | undo log |
| **一致性 (C)** | 事务执行前后数据一致性 | 约束、触发器 |
| **隔离性 (I)** | 并发事务互不干扰 | MVCC、锁 |
| **持久性 (D)** | 提交后数据永久保存 | redo log |

### 2.2 事务隔离级别

| 隔离级别 | 脏读 | 不可重复读 | 幻读 | 实现 |
|---------|------|-----------|------|------|
| **读未提交** | ✓ | ✓ | ✓ | 无 |
| **读已提交 (RC)** | ✗ | ✓ | ✓ | MVCC |
| **可重复读 (RR)** | ✗ | ✗ | ✓（部分）| MVCC + 间隙锁 |
| **串行化** | ✗ | ✗ | ✗ | 锁 |

**MySQL 默认 RR，Oracle 默认 RC**

**并发问题解释**：
- **脏读**：读到其他事务未提交的数据
- **不可重复读**：同一事务内两次读取结果不同（被其他事务修改）
- **幻读**：同一事务内两次查询，行数不同（被其他事务插入/删除）

### 2.3 MVCC 多版本并发控制

**实现原理**：
- 每行记录有两个隐藏列：`DB_TRX_ID`（事务 ID）、`DB_ROLL_PTR`（回滚指针）
- undo log 形成版本链
- ReadView 判断数据可见性

**ReadView 结构**：
```text
creator_trx_id: 创建该 ReadView 的事务 ID
m_ids: 未提交事务 ID 列表
min_trx_id: m_ids 中最小的事务 ID
max_trx_id: 下一个要分配的事务 ID
```

**可见性判断规则**：
1. `DB_TRX_ID == creator_trx_id`：可见（自己修改的）
2. `DB_TRX_ID < min_trx_id`：可见（已提交）
3. `DB_TRX_ID >= max_trx_id`：不可见（将来事务）
4. `min_trx_id <= DB_TRX_ID < max_trx_id`：
  - 在 m_ids 中：不可见（未提交）
  - 不在 m_ids 中：可见（已提交）

**RC vs RR 的 ReadView 差异**：
- RC：每次查询都生成新的 ReadView
- RR：事务开始时生成 ReadView，一直使用

### 2.4 锁机制

#### 锁类型

| 锁类型 | 说明 | 兼容性 |
|--------|------|--------|
| **共享锁 (S)** | 读锁 | S-S 兼容，S-X 冲突 |
| **排他锁 (X)** | 写锁 | X-X、X-S 都冲突 |

#### 锁粒度

| 粒度 | 说明 | 开销 |
|------|------|------|
| **行锁** | 锁定单行 | 小 |
| **间隙锁 (Gap Lock)** | 锁定范围，防止插入 | 中 |
| **临键锁 (Next-Key Lock)** | 行锁 + 间隙锁 | 中 |
| **表锁** | 锁定整张表 | 大 |
| **意向锁** | 表级锁的标记 | 小 |

**间隙锁解决幻读**：
```sql
-- 事务 A
SELECT * FROM user WHERE age > 20 FOR UPDATE;
-- 会锁定 (20, +∞) 的间隙，阻止其他事务插入 age > 20 的记录
```

#### 死锁的产生与解决

**死锁产生的四个条件**：
1. 互斥条件
2. 持有并等待
3. 不可剥夺
4. 循环等待

**检测与解决**：
- 超时检测：`innodb_lock_wait_timeout`
- 死锁检测：等待图算法，自动回滚其中一个事务

**预防策略**：
- 按固定顺序获取锁
- 减少事务持有锁的时间
- 使用乐观锁（版本号）替代悲观锁

## 三、MySQL 日志

### 3.1 三种日志对比

| 日志 | 作用 | 写入时机 | 特点 |
|------|------|---------|------|
| **undo log** | 回滚、MVCC | 事务执行时 | 逻辑日志，用于撤销修改 |
| **redo log** | 崩溃恢复 | 事务提交时 | 物理日志，保证持久性 |
| **binlog** | 备份、主从复制 | 事务提交时 | 逻辑日志，SQL 语句 |

### 3.2 redo log 详解

**为什么需要 redo log？**
- 直接写磁盘（随机 IO）性能差
- 先写 redo log（顺序 IO），再异步刷盘
- 崩溃后根据 redo log 恢复

**WAL（Write-Ahead Logging）原则**：
- 先写日志，再写磁盘
- 日志顺序写，性能高

**redo log 结构**：
```text
+--------+--------+--------+--------+
| 表空间号 | 页号   | 偏移量 | 修改值 |
+--------+--------+--------+--------+
```

**刷盘策略（innodb_flush_log_at_trx_commit）**：
| 值 | 行为 | 性能 | 安全性 |
|---|------|------|--------|
| 0 | 每秒刷盘 | 最高 | 最低（可能丢 1 秒数据）|
| 1 | 每次提交刷盘 | 最低 | 最高 |
| 2 | 每次提交写 OS 缓存 | 中 | 中（系统崩溃可能丢）|

### 3.3 binlog 详解

**binlog 格式**：
| 格式 | 说明 | 优点 | 缺点 |
|------|------|------|------|
| **STATEMENT** | 记录 SQL 语句 | 日志小 | 某些语句可能不一致 |
| **ROW** | 记录行变化 | 精确 | 日志大 |
| **MIXED** | 混合使用 | 平衡 | - |

**主从复制原理**：
```text
主库                              从库
  │                                │
  ├── binlog (dump thread) ───────>│
  │                                ├── I/O thread 写入 relay log
  │                                ├── SQL thread 重放
  │                                └── 数据同步
```

**复制模式**：
- 异步复制：主库不等待从库（性能高，可能丢数据）
- 半同步复制：至少一个从库确认（平衡）
- 同步复制：所有从库确认（一致性高，性能低）

## 四、性能优化

### 4.1 SQL 优化

**EXPLAIN 分析**：
```sql
EXPLAIN SELECT * FROM user WHERE name = '张三';
```

| 字段 | 说明 | 注意 |
|------|------|------|
| type | 访问类型 | system > const > eq_ref > ref > range > index > ALL |
| key | 使用的索引 | - |
| rows | 扫描行数 | 越小越好 |
| Extra | 额外信息 | Using index（覆盖索引）、Using filesort（需优化）|

**常见优化手段**：
1. **索引优化**：建立合适的索引，避免全表扫描
2. **覆盖索引**：避免回表查询
3. **最左前缀**：联合索引正确使用
4. **避免 SELECT ***：只查询需要的列
5. **批量操作**：减少交互次数
6. **分页优化**：深分页使用延迟关联

**深分页优化**：
```sql
-- 低效
SELECT * FROM user LIMIT 1000000, 10;

-- 优化：使用覆盖索引 + 延迟关联
SELECT * FROM user u
JOIN (SELECT id FROM user LIMIT 1000000, 10) tmp
ON u.id = tmp.id;
```

### 4.2 表结构设计

**范式设计**：
| 范式 | 要求 | 适用场景 |
|------|------|---------|
| **1NF** | 列不可再分 | 基础要求 |
| **2NF** | 非主键完全依赖主键 | 消除部分依赖 |
| **3NF** | 消除传递依赖 | 常规设计 |
| **反范式设计** | 适当冗余 | 读多写少，提高查询性能 |

**分库分表**：
| 方案 | 适用场景 | 策略 |
|------|---------|------|
| **垂直分表** | 字段过多 | 冷热字段分离 |
| **垂直分库** | 业务耦合低 | 按业务拆分 |
| **水平分表** | 单表数据量大 | 按 ID/时间取模 |
| **水平分库** | 单库压力大 | 同水平分表 |

**分片键选择原则**：
- 查询频率高
- 数据分布均匀
- 避免跨分片查询

## 五、Redis

### 5.1 数据结构与使用场景

| 数据结构 | 底层实现 | 适用场景 |
|---------|---------|---------|
| **String** | SDS | 缓存、计数器、分布式锁 |
| **List** | QuickList | 消息队列、时间线 |
| **Hash** | HashTable/Ziplist | 对象存储、购物车 |
| **Set** | HashTable/Intset | 去重、交集并集 |
| **ZSet** | SkipList + HashTable | 排行榜、延迟队列 |
| **Bitmap** | String | 签到、在线状态 |
| **HyperLogLog** | 稀疏/密集矩阵 | UV 统计 |
| **Geo** | ZSet | 地理位置 |
| **Stream** | Radix Tree | 消息队列（Kafka 风格）|

### 5.2 Redis 为什么快？

1. **内存存储**：纯内存操作
2. **单线程**：避免线程切换开销（6.0+ 多线程 IO）
3. **高效数据结构**：SDS、跳表、压缩列表等
4. **IO 多路复用**：epoll 处理高并发连接
5. **RESP 协议**：简单高效的文本协议

### 5.3 持久化机制

| 方式 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| **RDB** | 定时快照 | 恢复快，文件紧凑 | 可能丢数据 |
| **AOF** | 记录写命令 | 数据安全 | 文件大，恢复慢 |
| **混合** | RDB + AOF | 平衡 | - |

**AOF 重写**：
- 创建当前数据集的最小命令集
- 减少 AOF 文件大小
- fork 子进程进行，不阻塞主进程

### 5.4 缓存问题与解决方案

#### 缓存穿透
**问题**：查询不存在的数据，绕过缓存直达数据库
**解决**：
- 布隆过滤器
- 缓存空值（设置短过期时间）

#### 缓存击穿
**问题**：热点 key 过期，大量请求直达数据库
**解决**：
- 互斥锁（只允许一个线程重建缓存）
- 逻辑过期（不设置 TTL，通过逻辑时间判断）

#### 缓存雪崩
**问题**：大量 key 同时过期，数据库压力剧增
**解决**：
- 过期时间加随机值
- 多级缓存
- 熔断降级

### 5.5 高可用架构

| 模式 | 原理 | 特点 |
|------|------|------|
| **主从复制** | 一主多从 | 读写分离，数据冗余 |
| **哨兵 (Sentinel)** | 监控主从，自动故障转移 | 自动高可用 |
| **集群 (Cluster)** | 数据分片，多主多从 | 水平扩展，去中心化 |

**Redis Cluster 分片**：
- 16384 个槽位（slot）
- 每个节点负责一部分槽
- 通过 CRC16(key) % 16384 计算槽位

## 六、分布式数据库

### 6.1 分布式事务

| 方案 | 原理 | 特点 |
|------|------|------|
| **2PC** | 两阶段提交 | 同步阻塞，单点故障 |
| **3PC** | 三阶段提交 | 改进 2PC，仍有阻塞 |
| **TCC** | Try-Confirm-Cancel | 业务层实现，补偿机制 |
| **Saga** | 长事务拆分 | 最终一致性 |
| **本地消息表** | 异步确保 | 最终一致性 |
| **Seata AT** | 自动补偿 | 对业务侵入小 |

### 6.2 分布式 ID 生成

| 方案 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| **UUID** | 随机生成 | 简单 | 无序，存储大 |
| **雪花算法** | 时间戳 + 机器 ID + 序列号 | 趋势递增 | 依赖时钟 |
| **数据库自增** | 数据库分配 | 简单 | 单点瓶颈 |
| **号段模式** | 批量获取 ID | 高性能 | 需维护 |

**雪花算法结构**：
```text
0 | 41 bit 时间戳 | 10 bit 机器 ID | 12 bit 序列号
```
