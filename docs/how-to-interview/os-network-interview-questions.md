# 操作系统与网络面试题精编

> 本文整理了操作系统和计算机网络相关的面试高频问题，涵盖进程线程、内存管理、网络协议、IO 模型等核心知识点。

---

## 一、进程与线程

### 1.1 进程 vs 线程

| 特性 | 进程 | 线程 |
|------|------|------|
| **定义** | 资源分配的基本单位 | CPU 调度的基本单位 |
| **内存空间** | 独立地址空间 | 共享进程地址空间 |
| **通信方式** | IPC（管道、消息队列、共享内存等）| 直接读写共享变量 |
| **切换开销** | 大（需切换页表）| 小（仅需切换寄存器）|
| **创建销毁** | 慢 | 快 |
| **安全性** | 一个崩溃不影响其他 | 一个崩溃可能导致整个进程崩溃 |

### 1.2 进程状态与转换

```
        创建
          ↓
      ┌───────┐
      │ 就绪 │ ←──────┐
      └───┬───┘        │
    调度  │            │ 时间片用完
          ↓            │
      ┌───────┐        │
      │ 运行 │─────────┘
      └───┬───┘
    等待  │
    IO    ↓
      ┌───────┐
      │ 阻塞 │
      └───┬───┘
    IO完成 │
          ↓
        终止
```

### 1.3 进程间通信（IPC）

| 通信方式 | 特点 | 适用场景 |
|---------|------|---------|
| **管道 (Pipe)** | 单向字节流，有亲缘关系 | 父子进程通信 |
| **命名管道 (FIFO)** | 无亲缘关系限制 | 无关进程通信 |
| **消息队列** | 消息为单位，可 selective read | 离散消息传递 |
| **共享内存** | 最快，需同步机制 | 大量数据共享 |
| **信号量** | 用于同步 | 配合共享内存使用 |
| **Socket** | 可跨网络 | 分布式通信 |
| **信号** | 异步通知 | 事件通知 |

**共享内存为什么最快？**
- 不需要内核介入数据拷贝
- 直接读写同一块物理内存
- 仅需一次系统调用建立映射

### 1.4 线程同步机制

| 同步方式 | 特点 | 效率 |
|---------|------|------|
| **互斥锁 (Mutex)** | 独占访问 | 中 |
| **读写锁 (RWLock)** | 读共享写独占 | 读多时高 |
| **自旋锁 (Spinlock)** | 忙等待 | 短临界区高 |
| **条件变量** | 等待/通知 | - |
| **信号量** | 计数器，允许多个 | - |
| **原子操作** | 硬件支持 | 最高 |

**自旋锁 vs 互斥锁**：
- 自旋锁：线程不休眠，循环等待（适合短临界区）
- 互斥锁：线程阻塞，让出 CPU（适合长临界区）

### 1.5 线程池设计

**为什么使用线程池？**
- 减少线程创建/销毁开销
- 控制并发线程数量
- 便于线程管理

**线程池核心参数**：
```cpp
class ThreadPool {
    size_t coreThreads;      // 核心线程数
    size_t maxThreads;       // 最大线程数
    size_t workQueueSize;    // 任务队列大小
    size_t keepAliveTime;    // 空闲线程存活时间
    RejectPolicy policy;     // 拒绝策略
};
```

**拒绝策略**：
- Abort：直接抛出异常
- CallerRuns：由调用线程执行任务
- Discard：静默丢弃
- DiscardOldest：丢弃队列最老任务

## 二、内存管理

### 2.1 虚拟内存

**为什么需要虚拟内存？**
1. **隔离性**：进程间内存隔离，互不干扰
2. **扩展性**：程序可使用比物理内存更大的地址空间
3. **简化编程**：每个进程都有统一的地址空间
4. **共享**：方便共享内存、动态库

**虚拟地址到物理地址转换**：
```text
虚拟地址 = 页号 + 页内偏移
              ↓ 页表查找
物理地址 = 物理页框号 + 页内偏移
```

### 2.2 页面置换算法

| 算法 | 思想 | 特点 |
|------|------|------|
| **FIFO** | 先进先出 | 简单，可能 Belady 异常 |
| **LRU** | 最近最少使用 | 性能好，需硬件支持 |
| **LFU** | 最少使用频率 | 考虑访问频率 |
| **Clock** | 二次机会 | LRU 的近似实现 |
| **OPT** | 最优（理论）| 需预知未来，不可实现 |

**Linux 内存管理策略**：
- 伙伴系统（Buddy System）：管理物理页框
- Slab 分配器：管理内核小对象

### 2.3 内核态 vs 用户态

| 特性 | 用户态 | 内核态 |
|------|--------|--------|
| **权限** | 受限 | 完全访问 |
| **内存访问** | 只能访问用户空间 | 可访问全部内存 |
| **指令执行** | 不能执行特权指令 | 可执行所有指令 |
| **切换触发** | 系统调用、中断、异常 | - |

**用户态到内核态的切换时机**：
- 系统调用（如 read/write）
- 中断（如时钟中断、IO 中断）
- 异常（如缺页异常）

**切换开销**：
- 保存/恢复上下文
- 切换页表
- 刷新 TLB
- 缓存失效

### 2.4 文件描述符

**什么是文件描述符？**
- 内核中打开文件表的索引
- 每个进程有独立的文件描述符表
- 0: stdin, 1: stdout, 2: stderr

**文件描述符与打开文件的关系**：
```text
进程 A:                   进程 B:
fd 3 ──→ 文件表项 ──┐     fd 4 ──→ 文件表项
fd 4 ──→ 文件表项 ──┼────→ inode (同一个文件)
                    └──→ inode
```

## 三、计算机网络基础

### 3.1 OSI 七层模型 vs TCP/IP 四层模型

| OSI 七层 | TCP/IP 四层 | 协议/设备 | 功能 |
|---------|------------|----------|------|
| 应用层 | | HTTP、FTP、DNS | 为应用程序提供服务 |
| 表示层 | 应用层 | SSL/TLS | 数据格式转换、加密 |
| 会话层 | | NetBIOS | 会话管理 |
| 传输层 | 传输层 | TCP、UDP | 端到端通信 |
| 网络层 | 网络层 | IP、ICMP、ARP | 路由寻址 |
| 数据链路层 | 链路层 | 以太网、WiFi | 帧传输、MAC 寻址 |
| 物理层 | | 网线、光纤 | 比特流传输 |

**路由器和交换机工作在哪层？**
- 路由器：网络层（基于 IP 地址转发）
- 交换机：数据链路层（基于 MAC 地址转发）

### 3.2 TCP vs UDP

| 特性 | TCP | UDP |
|------|-----|-----|
| **连接** | 面向连接 | 无连接 |
| **可靠性** | 可靠传输 | 不可靠 |
| **顺序** | 保证顺序 | 不保证 |
| **拥塞控制** | 有 | 无 |
| **流量控制** | 有 | 无 |
| **头部开销** | 20 字节 | 8 字节 |
| **适用场景** | HTTP、FTP、邮件 | 视频流、DNS、游戏 |

### 3.3 TCP 三次握手与四次挥手

**三次握手**：
```json
客户端                    服务端
   │     SYN(seq=x)      │
   │ ──────────────────> │
   │                     │
   │   SYN(seq=y,ack=x+1)│
   │ <────────────────── │
   │                     │
   │     ACK(ack=y+1)    │
   │ ──────────────────> │
   │                     │
   [ESTABLISHED]      [ESTABLISHED]
```

**为什么需要三次握手？**
- 确认双方发送和接收能力正常
- 防止历史重复连接初始化

**四次挥手**：
```text
客户端                    服务端
   │     FIN(seq=u)      │
   │ ──────────────────> │
   │   ACK(ack=u+1)      │
   │ <────────────────── │
   │                     │ [CLOSE_WAIT]
   │   FIN(seq=w,ack=u+1)│
   │ <────────────────── │
   │     ACK(ack=w+1)    │
   │ ──────────────────> │
   │                     │
   │ [TIME_WAIT]      [CLOSED]
   │ (2MSL 后关闭)       │
```

**为什么 TIME_WAIT 需要 2MSL？**
1. 确保最后一个 ACK 被对方收到（超时可重传）
2. 等待网络中残留的数据包消失，防止新连接收到旧数据

### 3.4 TCP 可靠传输机制

**1. 序列号与确认应答 (ACK)**
- 每个字节都有序列号
- 累计确认：ACK 表示已收到该序号之前的所有数据

**2. 超时重传**
- RTO（重传超时时间）根据 RTT 动态计算
- 超时未收到 ACK 则重传

**3. 快速重传**
- 收到 3 个重复 ACK 立即重传
- 不等待超时

**4. 滑动窗口（流量控制）**
```text
发送方窗口：
已发送已确认 | 已发送未确认 | 可发送 | 不可发送
     ↑              ↑           ↑
   LastByteAcked  LastByteSent  LastByteWritten

接收方窗口：
已接收 | 未接收（窗口内）| 未接收（窗口外）
  ↑            ↑
LastByteRead  LastByteRcvd
```

**5. 拥塞控制**

| 阶段 | 算法 | 特点 |
|------|------|------|
| 慢启动 | Slow Start | 指数增长（cwnd 翻倍）|
| 拥塞避免 | Congestion Avoidance | 线性增长 |
| 快速恢复 | Fast Recovery | 收到 3 个 dup ACK |

**拥塞控制算法**：
- Reno：传统算法
- CUBIC：Linux 默认，适合高带宽延迟网络
- BBR：Google 开发，基于带宽和 RTT

### 3.5 TCP 粘包与拆包

**什么是粘包？**
- TCP 是字节流协议，不保留消息边界
- 多条消息可能被合并成一个数据包发送

**解决方案**：
1. **固定长度**：每个消息固定大小
2. **分隔符**：如 `\n` 分隔
3. **长度前缀**：消息头包含长度信息
4. **特殊协议**：如 protobuf、HTTP 的 Content-Length

## 四、IO 模型

### 4.1 五种 IO 模型

| 模型 | 特点 | 效率 |
|------|------|------|
| **阻塞 IO** | 调用后等待完成 | 低 |
| **非阻塞 IO** | 立即返回，轮询检查 | 中 |
| **IO 多路复用** | 一个线程管理多个连接 | 高 |
| **信号驱动 IO** | 事件通知 | 中 |
| **异步 IO** | 完全异步，完成回调 | 最高 |

### 4.2 select、poll、epoll 对比

| 特性 | select | poll | epoll |
|------|--------|------|-------|
| **文件描述符限制** | 1024（FD_SETSIZE）| 无限制 | 无限制 |
| **触发方式** | 水平触发 | 水平触发 | 水平/边缘触发 |
| **时间复杂度** | O(n) | O(n) | O(1) |
| **数据拷贝** | 每次调用拷贝 fd 集合 | 每次调用拷贝 fd 数组 | 内核维护，epoll_ctl 时拷贝 |
| **实现方式** | 轮询所有 fd | 轮询所有 fd | 回调通知 |

**epoll 高效的原因**：
1. 使用红黑树存储 fd，O(log n) 增删
2. 使用就绪链表存储就绪 fd，O(1) 获取
3. 回调机制：fd 就绪时回调 `ep_poll_callback` 添加到就绪链表
4. 无需每次拷贝 fd 集合到内核

**LT（水平触发）vs ET（边缘触发）**：
| 特性 | LT | ET |
|------|-----|-----|
| **触发条件** | 可读/可写时一直触发 | 状态变化时触发一次 |
| **编程难度** | 简单 | 复杂（需一次性读完）|
| **效率** | 可能重复触发 | 更高 |
| **数据丢失风险** | 低 | 需确保读完全部数据 |

### 4.3 Reactor vs Proactor

| 特性 | Reactor | Proactor |
|------|---------|----------|
| **模式** | 同步 IO | 异步 IO |
| **通知时机** | IO 就绪时通知 | IO 完成时通知 |
| **数据处理** | 应用层读取 | 系统读取后通知应用 |
| **实现** | select/poll/epoll | Windows IOCP、Linux AIO |
| **复杂度** | 较低 | 较高 |

**Reactor 组件**：
- Reactor：事件分发器
- Acceptor：连接接收器
- Handler：事件处理器

## 五、网络编程实践

### 5.1 Socket 编程流程

**服务端**：
```c
socket() → bind() → listen() → accept() → recv/send → close()
```

**客户端**：
```c
socket() → connect() → send/recv → close()
```

**关键参数**：
- `listen(backlog)`：半连接队列 + 全连接队列长度
- `SO_REUSEADDR`：地址复用，快速重启
- `SO_REUSEPORT`：端口复用，多进程负载均衡

### 5.2 高并发服务器设计

**C10K 问题解决方案**：

| 方案 | 特点 | 适用 |
|------|------|------|
| **多进程** | 每个连接一个进程 | 连接数少，计算密集 |
| **多线程** | 每个连接一个线程 | 连接数中等 |
| **IO 多路复用 + 单线程** | 事件驱动 | 高并发，IO 密集 |
| **IO 多路复用 + 线程池** | 主线程 accept，工作线程处理 | 高并发，部分计算 |
| **多 Reactor** | 主 Reactor accept，子 Reactor 处理 IO | 极高并发 |

### 5.3 服务器最大连接数限制

**理论限制**：
- 端口数量：65535（实际可用约 64511）
- 文件描述符：ulimit -n 设置
- 内存：每个连接占用内存

**突破限制**：
- 多 IP：每个 IP 65535 个端口
- 长连接复用：HTTP keep-alive
- 连接池：客户端复用连接

## 六、HTTP 协议

### 6.1 HTTP 版本演进

| 版本 | 特性 |
|------|------|
| HTTP/1.0 | 短连接，每次请求新建 TCP |
| HTTP/1.1 | 长连接（Keep-Alive）、管道化、分块传输 |
| HTTP/2 | 二进制分帧、多路复用、头部压缩、服务器推送 |
| HTTP/3 | 基于 QUIC（UDP），0-RTT 握手 |

### 6.2 HTTP 状态码

| 类别 | 状态码 | 含义 |
|------|--------|------|
| 2xx | 200 | OK |
| | 204 | No Content |
| 3xx | 301 | Moved Permanently |
| | 302 | Found（临时重定向）|
| | 304 | Not Modified（缓存有效）|
| 4xx | 400 | Bad Request |
| | 403 | Forbidden |
| | 404 | Not Found |
| 5xx | 500 | Internal Server Error |
| | 502 | Bad Gateway |
| | 503 | Service Unavailable |

### 6.3 HTTPS 加密过程

**TLS 握手过程**：
```text
1. 客户端发送：ClientHello（支持的加密套件、随机数）
2. 服务端发送：ServerHello（选定的加密套件、随机数）、证书
3. 客户端验证证书，生成 Pre-Master Secret，用公钥加密发送
4. 双方根据随机数和 Pre-Master Secret 生成会话密钥
5. 后续通信使用对称加密
```

**为什么不用非对称加密传输数据？**
- 非对称加密计算开销大（RSA 比 AES 慢 100-1000 倍）
- 对称加密适合大量数据传输

## 七、网络排查工具

### 7.1 常用命令

| 命令 | 用途 |
|------|------|
| `ping` | 测试连通性 |
| `traceroute` | 追踪路由路径 |
| `netstat` | 查看网络连接、端口 |
| `ss` | 比 netstat 更快（替代方案）|
| `tcpdump` | 抓包分析 |
| `nc` | 网络调试、端口扫描 |
| `curl` | HTTP 请求测试 |
| `nmap` | 端口扫描 |

### 7.2 TCPDump 使用示例

```bash

# 抓取所有经过 eth0 的包
tcpdump -i eth0

# 抓取特定端口的包
tcpdump port 80

# 抓取特定主机
tcpdump host 192.168.1.1

# 保存到文件
tcpdump -w capture.pcap

# 读取文件
tcpdump -r capture.pcap
```
