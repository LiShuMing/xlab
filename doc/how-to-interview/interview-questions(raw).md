- 你提到了brk和mmap这两种分配方式，能详细讲讲吗？什么时候选择brk，什么时候选择mmap？内存阈值是多少？
    - 取决于分配的内存块大小
    - **小内存分配 (`brk`)：** 当 `malloc` 请求的内存较小且适合放在堆中时，`glibc` 会优先使用 `brk` 来扩展堆的大小并满足内存分配需求。
    - **大内存分配 (`mmap`)：** 当请求的内存较大时，为了避免堆的碎片化以及更好地管理内存，`glibc` 会使用 `mmap` 来直接分配内存块。
    - **内存释放：** 使用 `brk` 分配的内存只有当整个堆顶段都空闲时，`sbrk` 才能将其释放回操作系统。使用 `mmap` 分配的内存则可以通过 `munmap` 直接释放回操作系统。
- 你提到了使用brk分配内存的时候，free的时候并不会立即归还给系统，为什么要做这样的事？
    - **性能优化**
        - **减少系统调用开销：** 每次调整堆大小（增加或减少）都需要进行一次系统调用，而系统调用相对于用户态操作而言是昂贵的。频繁调整堆大小会导致系统性能下降。因此，内存管理库通常会将释放的内存保留在进程中，以减少不必要的 `brk` 调用。
        - **提高分配效率：** 保留已释放的内存可以加快后续的内存分配。如果立即释放内存，下一次分配时可能需要再次调用 `brk` 或 `sbrk` 扩展堆顶，而保留内存则可以直接复用已有的空闲空间，从而避免系统调用的开销。
    - **减少堆碎片化**
        - **堆碎片化问题：** 如果每次释放的内存都立即归还给操作系统，堆可能会变得高度碎片化，这使得分配大块连续内存变得困难。通过保留已释放的内存，内存管理库可以更好地管理和合并空闲块，减少碎片化的可能性。
        - **延迟归还：** 一般情况下，只有当堆顶的整个大块空闲内存可以释放时，`sbrk` 才会将其归还给操作系统。这可以减少碎片化的影响，因为堆顶的连续空闲块更容易被系统回收。
    - **内存复用**
        - **常见分配模式的复用：** 在许多应用程序中，内存分配和释放具有一定的模式（例如，特定大小的内存块反复分配和释放）。保留内存块可以更有效地复用这些内存，从而避免频繁的系统调用。
        - **池化策略：** 内存管理库通常会实现一些内存池化策略，以应对常见的分配模式。保留内存块可以使得这些策略更加高效，例如将同一大小的内存块放入一个空闲列表中，等待后续分配直接复用。
    - **延迟释放策略**
        - **动态调整堆大小：** 大多数内存管理库采用延迟释放策略，只有在内存需求明显下降时才会调整堆大小。这种策略通过观察一段时间内内存的使用模式来决定是否归还内存给操作系统，从而在性能和内存利用率之间取得平衡。
        - **避免不必要的释放和分配：** 如果释放的内存很快就会再次分配，那么立即归还给操作系统显然是不经济的。延迟释放可以避免频繁的分配和释放操作，减少性能损耗。
    - 如果用malloc频繁的申请小块的内存，会出现什么问题？
        - **内存碎片化**
            - **内部碎片化**: 由于 `malloc` 通常会为每次分配预留一些额外的元数据（如块大小、状态等），小块内存的分配和释放会导致堆中产生许多大小不一的空闲块。这些空闲块可能无法被有效利用，导致内存的内部碎片化，即分配的内存块中有许多未使用的空间。
            - **外部碎片化**: 由于 `malloc` 无法保证分配的内存块在堆中是连续的，频繁的小块分配和释放会使得堆空间出现许多不连续的、被分割的内存区域。这种外部碎片化可能导致在后续需要分配较大内存块时，尽管总内存足够，但无法找到足够大的连续内存块，导致分配失败。
        - **性能问题**
            - **频繁的系统调用**: 虽然 `malloc` 在管理小块内存时通常会尽量复用现有的空闲块，但随着时间的推移，系统可能会因为碎片化而需要频繁调用 `sbrk` 或 `mmap` 来分配新的堆空间或创建新的内存映射。这会增加系统调用的开销，导致性能下降。
            - **管理开销增加**: 每次 `malloc` 和 `free` 操作都涉及内存管理的数据结构（如空闲链表、内存池等）的更新。频繁的小块分配会导致这些管理数据结构的频繁修改，进而增加程序的开销。

    - **适应性分配（Best-Fit, First-Fit, etc.）**

- 操作系统如何做线程调度的？
    - **调度队列**：操作系统维护一个或多个线程调度队列，其中包含了就绪执行的线程。**优先级**：线程通常被分配不同的优先级。调度器通常优先考虑高优先级的线程。
    - **调度策略**：
        - **先来先服务（FCFS）**：按照线程到达的顺序进行调度。
        - **短作业优先（SJF）**：优先调度预计执行时间短的线程。
        - **轮转（Round-Robin, RR）**：所有线程被分配时间片，轮流执行。
        - **多级队列**：线程根据优先级或其他属性被分配到不同的队列中。
    - **抢占式调度**：在抢占式调度中，调度器可以中断正在运行的线程，并将 CPU 资源分配给另一个线程。
    - **协作式调度**：在协作式调度中，线程自行放弃 CPU，通常是通过调用 `yield` 函数或其他同步机制。
    - **调度算法**：调度器使用不同的算法来决定线程的调度顺序，如完全公平调度算法（CFS）等。
    - **负载均衡**：在多处理器或多核系统中，操作系统还需要考虑线程在不同处理器之间的分布，以实现负载均衡。
    - **线程绑定**：在某些系统中，线程可以被绑定到特定的处理器上，以减少上下文切换和缓存失效。
    - **实时调度**：对于实时系统，调度器需要保证高优先级的线程能够及时获得 CPU 资源。
- 现在有一个多线程的程序，他们同时去修改一个int类型的全局变量，那这个时候会有线程安全问题吗？为什么会有线程安全问题？比如多个线程同时对i++，会有什么问题？（只加了一次）那应该怎么避免这种情况的发生呢？（单例模式）你提到了全局变量使用单例模式来替代，那具体应该怎么做呢？（静态局部对象）
    - 现成安全的原因：
        1. **竞态条件（Race Condition）**：如果两个或多个线程能够同时读取同一个变量，然后至少有一个线程会写入这个变量，那么这个变量就是不稳定的。这种情况下，变量的最终状态取决于线程的执行顺序，这是不可预测的。
        2. **原子性问题**：在大多数现代 CPU 上，`i++` 操作不是一个原子操作。它可能涉及多个步骤：读取变量的值、增加值、写回新值。如果两个线程同时执行 `i++`，一个线程可能会读取到另一个线程已经增加但尚未写回的中间值。
        3. **可见性问题**：一个线程对变量的修改可能对其他线程不可见，因为修改可能被缓存在寄存器或线程的局部内存中。
    - 避免线程安全问题
        1. **互斥锁（Mutex）**：使用互斥锁来确保每次只有一个线程可以修改变量。在访问变量之前加锁，在访问结束后解锁。
        2. **原子操作**：使用原子类型和原子操作，如 C++11 中的 `<atomic>` 库提供的原子变量和操作。
        3. **读写锁**：如果读操作远多于写操作，可以使用读写锁来允许多个线程同时读取，但写入时需要独占访问。
        4. **条件变量**：与互斥锁结合使用，可以协调线程间的工作，确保在特定条件下才进行写操作。
        5. **单例模式**：单例模式确保一个类只有一个实例，并提供一个全局访问点。这不直接解决线程安全问题，但可以通过在单例的实现中加入线程安全措施。
        6. **静态局部对象**：在 C++ 中，静态局部对象在程序的生命周期内只初始化一次，并且初始化是线程安全的。但是，这并不适用于全局变量的线程安全问题。

- 能讲讲CPU的缓存吗？它和内存有什么区别？那你能讲讲如何做才能避免缓存不同步的问题吗？（MESI协议）能讲讲这个协议吗？
    - 避免缓存不同步的问题：
        - 缓存一致性问题通常发生在多处理器系统中，不同的CPU核心可能有自己的缓存，并且共享同一块主内存。为了维护缓存之间的一致性，使用了一系列协议，其中最著名的是MESI协议。
    - MESI协议：MESI协议是一种缓存一致性协议，用于在多处理器系统中同步各个处理器的缓存。它基于以下四种状态：
        1. **Modified（修改）**：缓存行只在此处理器的缓存中，并且已经被修改。它与主内存中的版本不同。
        2. **Exclusive（独占）**：缓存行只在此处理器的缓存中，但尚未被修改。它可以被安全地写回主内存而不影响其他处理器。
        3. **Shared（共享）**：缓存行可能被多个处理器缓存，所有副本都是干净的（与主内存一致）。
        4. **Invalid（无效）**：缓存行不再包含有效的数据，可能是因为数据已经被其他处理器修改或写回。
- DNS解析的过程是怎样的？DNS向其他几级服务器查询的时候，用的什么协议？
    - 查询过程
        1. **客户端查询**：当用户在浏览器中输入一个网址或应用程序需要解析域名时，DNS解析过程开始。
        2. **本地DNS缓存检查**：首先，系统会检查本地DNS缓存（如操作系统的DNS解析缓存或路由器的DNS缓存），看是否已经有了该域名的解析结果。如果有，直接返回结果。
        3. **递归查询**：
            - 如果本地缓存中没有结果，系统会向配置的本地DNS服务器（通常是ISP提供的）发送查询请求。
            - 本地DNS服务器同样会先检查自己的缓存，如果有缓存结果，直接返回。
        4. **迭代查询**：
            - 如果本地DNS服务器没有缓存结果，它会开始一个迭代查询过程，向更高级别的DNS服务器查询。
            - 首先查询根DNS服务器（.），获得负责顶级域（如.com、.org）的顶级域DNS服务器的地址。
        5. **顶级域服务器查询**：然后，本地DNS服务器向相应的顶级域DNS服务器查询，获得负责特定域（如example.com）的权威DNS服务器的地址。
        6. **权威DNS服务器查询**：最后，本地DNS服务器向权威DNS服务器查询，获取域名对应的IP地址。
        7. **返回结果**：一旦获得IP地址，本地DNS服务器将其返回给客户端，并可能缓存这个结果以备后续使用。
        8. **重复查询**：如果查询过程中某个DNS服务器没有相应记录，可能需要向备用的DNS服务器或更高级别的DNS服务器重复查询过程。
    - **DNS协议**：基于UDP（用户数据报协议）的53号端口，用于快速、简单的查询和响应。
    - **TCP（传输控制协议）**：当返回的DNS数据包大小超过UDP报文的最大长度（通常为512字节）时，DNS可以使用TCP的53号端口来传输更大的数据包。

- new和malloc的区别
    - **类型安全**：`new` 是类型安全的，它知道分配对象的类型，并且会自动调用构造函数。`malloc` 不是类型安全的，它只分配内存，不涉及类型信息，也不会调用构造函数。
    - **构造函数和析构函数**：使用 `new` 分配的对象会自动调用构造函数，使用 `delete` 释放时会自动调用析构函数。使用 `malloc` 分配的内存不会自动调用构造函数或析构函数，需要手动构造和析构（如果对象是多态类型）。
    - **内存分配失败**：`new` 在内存分配失败时会抛出一个 `std::bad_alloc` 异常。`malloc` 在内存分配失败时返回 `nullptr`。
    - **内存大小**：`new` 分配的内存大小包括对象的实际大小加上一些额外的内存用于存储类型信息和调试信息（在某些实现中）。`malloc` 仅分配请求的大小。**内存对齐**：`new` 保证分配的内存满足类型对齐的要求。

- 进程和线程的同步方式，哪些同步方式可以用在线程中，哪些同步方式可以用在进程中而不能用在线程中？同步方式哪种效率最高，为什么
    - 线程间同步方式：
        - **互斥锁（Mutex）**：用于确保多个线程不会同时进入临界区。`std::mutex` 是 C++ 中常用的互斥锁。
        - **条件变量（Condition Variable）**：允许线程在某些条件不满足时等待，并在条件满足时被唤醒。通常与互斥锁结合使用。
        - **读写锁（Read-Write Lock）**：允许多个线程同时读取资源，但在写操作时独占资源。`std::shared_mutex` 是 C++ 中的读写锁。
        - **自旋锁（Spinlock）**：类似于互斥锁，但在锁不可用时会忙等待（自旋），适用于锁的持有时间很短的场景。
        - **原子操作（Atomic Operation）**：使用硬件级别的原子指令实现线程间的同步，而无需锁。`std::atomic` 提供了原子操作的支持。
    - 进程间同步方式：
        1. **管道（Pipes）**：用于进程间通信，不适合线程间通信。
        2. **消息队列（Message Queues）**：用于进程间的消息传递，不适用于线程。
        3. **共享内存（Shared Memory）**：虽然可以用于线程，但它的主要设计是为了方便进程间通信。
        4. **套接字（Sockets）**：主要用于进程间网络通信，不适合线程内同步。
    - 性能考虑：
        - **原子操作**是效率最高的同步方式，尤其是在多线程环境中。这是因为原子操作直接由硬件支持，不需要进行复杂的上下文切换或陷入内核，因此它们的开销最低。
            - **优点**：避免了锁机制带来的线程阻塞和上下文切换，效率高。
            - **局限性**：只能用于简单的数据操作，无法处理复杂的临界区逻辑。
        - **自旋锁**在锁定时间非常短的情况下，效率可能比传统的互斥锁高，因为它避免了线程的上下文切换。但如果自旋时间过长，会消耗大量 CPU 资源。
        - **互斥锁**在大多数情况下是效率较高且通用的同步方式，特别适用于线程间同步。当竞争少或锁持有时间短时，互斥锁的开销相对较小。
        - **信号量**在进程间同步中效率较高，特别适用于允许多个进程访问的场景。不过，由于进程间通信机制的开销，一般会比线程间同步的方式慢一些。

- 编译链接原理
    - 1. **预处理（Preprocessing）**预处理是编译的第一阶段，主要任务是处理源码中的预处理指令（通常以 `#` 开头，如 `#include`、`#define` 等）。预处理器的工作包括：
        - **宏展开**：将定义的宏替换为实际内容。
        - **头文件包含**：将 `#include` 的头文件内容插入到源代码中。
        - **条件编译**：根据条件编译指令（如 `#ifdef`、`#ifndef`）决定哪些代码需要编译。
        - **删除注释**：去除源代码中的注释，使其不影响后续编译。
    
    预处理的输出是一个“纯净”的 C 或 C++ 源代码文件，这些文件通常以 `.i` 为扩展名。
    
    - **编译（Compilation）**
        
        在编译阶段，预处理过的源代码文件（`.i` 文件）会被转换为中间语言，通常是汇编语言。这一过程包括以下步骤：
        
        - **词法分析**：将源代码分解为最小的语法单元，称为词法单元或记号（token），如关键字、标识符、操作符等。
        - **语法分析**：根据语言的语法规则，将词法单元组合成语法树（Syntax Tree）或抽象语法树（Abstract Syntax Tree，AST）。
        - **语义分析**：检查语法树的语义，确保变量声明、类型匹配、函数调用等方面的合法性。
        - **中间代码生成**：将语法树转换为中间代码，通常是一个与具体机器无关的中间表示。
        - **优化**：对中间代码进行优化，减少冗余计算，改进代码执行效率。
        - **目标代码生成**：将优化后的中间代码生成汇编代码。
        
        编译的输出通常是汇编代码文件，扩展名为 `.s`。
        
    - **汇编（Assembly）**
        
        汇编是将汇编代码（`.s` 文件）转换为机器语言（目标代码）。汇编器（Assembler）负责将每条汇编指令翻译成对应的机器指令，并生成可重定位目标文件（`.o` 或 `.obj` 文件）。目标文件包含：
        
        - **机器代码**：可直接执行的二进制指令。
        - **数据段**：程序的全局变量和常量。
        - **符号表**：记录符号（如函数名和变量名）及其在目标文件中的位置，用于链接阶段。
        - **重定位信息**：用于在链接阶段调整符号地址，确保程序能正确引用函数和变量。
    - **链接（Linking）**
        
        链接是编译过程的最后阶段，负责将多个目标文件及库文件（如 `.o`、`.a`、`.so`）组合成一个完整的可执行文件。链接器（Linker）的主要任务包括：
        
        - **符号解析**：链接器检查所有目标文件中的符号，确保每个符号（如函数或变量）都被正确定义和引用。未定义符号会导致链接错误。
        - **地址和重定位**：将所有符号分配实际的内存地址，并调整目标代码中的地址引用，使它们指向正确的位置。
        - **库的链接**：将目标文件所依赖的静态库或动态库中的代码合并到最终的可执行文件中。静态库在编译时直接链接到可执行文件中，而动态库（共享库）在运行时加载。
        - **生成可执行文件**：将所有目标代码和数据段按照内存布局规则组织在一起，生成最终的可执行文件（如 `.exe`、`.out`）。
    - 链接类型
        - **静态链接**：所有依赖的库在编译时被复制到可执行文件中，生成的可执行文件独立，不需要外部库文件。这种方式会增加可执行文件的大小，但在不同系统上运行时不需要考虑库的兼容性。
        - **动态链接**：可执行文件在运行时加载依赖的共享库，库的代码不被复制到可执行文件中。这种方式可以减少可执行文件的大小，但要求运行环境中必须存在所依赖的共享库。

    - tcp如何实现可靠性传输
        - **序列号（Sequence Number）**
            - 每个 TCP 段在传输时都会被分配一个序列号，序列号用于标识数据在整个字节流中的位置。接收方可以通过序列号确保按正确顺序接收数据，即使数据包在传输过程中乱序到达。
            - **初始序列号**：在 TCP 三次握手建立连接时，通信的双方都会各自生成一个初始序列号（ISN）。此后，每发送一个字节的数据，序列号会相应增加。
        - **确认机制（Acknowledgment, ACK**
            - 接收方在成功接收到数据后，会向发送方发送一个确认（ACK）消息。ACK 消息包含了一个确认号（Acknowledgment Number），该确认号表明接收方期望接收的下一个字节的序列号。也就是说，ACK 确认了所有在此之前的数据都已正确接收。
            - **累计确认**：TCP 的确认机制是累计确认的，即接收方发送的 ACK 是对所有已接收数据的确认，而不是逐个确认。
    - **重传机制（Retransmission）**
        - 为了防止数据丢失，TCP 实现了数据重传机制。发送方会在发送数据后启动一个定时器，如果在超时时间内没有收到相应的 ACK，则会重新发送相应的数据段。
        - **超时重传**：如果发送方在设定的超时时间内没有收到 ACK，则认为该段数据可能丢失，触发超时重传。
        - **快速重传**：当接收方发现某个数据包丢失时，可能会多次收到相同的 ACK（称为冗余 ACK）。当发送方连续收到三个相同的冗余 ACK 时，会立即重传丢失的数据段，而不必等待超时。
    - **流量控制（Flow Control）**
        - TCP 使用流量控制机制来避免发送方发送的数据超过接收方的处理能力。TCP 通过滑动窗口机制实现流量控制。
        - **滑动窗口**：发送方维护一个窗口，表示当前可以发送的字节范围。窗口大小由接收方根据其自身的缓冲区情况通过 `Window Size` 字段告知发送方。如果接收方的缓冲区接近满时，窗口大小将减小，迫使发送方减少发送数据量。
    - **拥塞控制（Congestion Control）**
        - TCP 实现了拥塞控制机制，以防止网络拥塞导致数据包丢失。TCP 拥塞控制包含四个主要算法：
        - **慢启动（Slow Start）**：在连接开始或网络恢复时，TCP 会以指数方式增加拥塞窗口（cwnd）的大小，以尽快找到网络的可用带宽。
        - **拥塞避免（Congestion Avoidance）**：当网络达到饱和时，拥塞窗口的增加速率从指数变为线性，以避免引发拥塞。
        - **快速重传和快速恢复（Fast Retransmit and Fast Recovery）**：在检测到丢包（通过冗余 ACK）时，快速重传机制会立即重传丢失的数据段，并且通过快速恢复机制避免触发慢启动，直接进入拥塞避免阶段。
    - **数据校验（Checksum）**
        - TCP 段包含一个校验和（Checksum）字段，用于检测数据在传输过程中是否发生错误。发送方计算整个 TCP 段（包括头部和数据部分）的校验和，并将结果填入校验和字段。接收方在接收数据后，重新计算校验和并与 TCP 段中的校验和字段进行比较，如果不匹配，则认为数据损坏并丢弃该段。
    - **重复数据检测**
        - TCP 能够检测并丢弃重复的段，这通常是由网络问题或重传引起的。当接收方接收到重复的数据段时，它会识别出这些段已经被接收并确认，并丢弃这些重复的数据段。
    - **丢包检测和处理**
    - 通过 ACK 和超时机制，TCP 可以检测到数据包的丢失。一旦检测到丢包，TCP 会通过重传机制重新发送丢失的数据段，确保数据的完整传输。

    - tcp连接为什么不能2次，关闭非得4次吗？
        - TCP为何不能2次：
            - **保证双向通信**：如果只用两次握手，客户端在发送 SYN 后直接开始传输数据，服务器可能会因为网络延迟或其他原因没有正确收到客户端的 SYN，导致通信失败。而三次握手确保双方都明确知道对方可以接收和发送数据，从而建立可靠的连接。
            - **避免历史连接误操作**：三次握手还能防止历史连接的旧 SYN 包导致的误连接。例如，一个失效的 SYN 包可能在网络中延迟到达，这时如果使用两次握手，服务器可能会错误地建立连接。而通过三次握手，客户端的 ACK 能确认此次连接是最新的，有效避免了这种情况。
        - 关闭为何要4次
            - **全双工关闭**：TCP 是一个全双工协议，意味着通信双方都可以同时发送和接收数据。为了确保双方的数据都传输完毕，连接的每一端都需要单独关闭发送和接收通道。这就是为什么需要两对 FIN 和 ACK，分别关闭发送和接收。
            - **半关闭状态**：当一方发送 FIN 后，它进入半关闭状态，仍然可以接收数据。这允许另一方在关闭连接前完成数据传输，确保所有数据都能正确接收。

        - **硬件效率**：大多数现代 CPU 都通过总线（如 32 位、64 位总线）来访问内存。这些总线一次可以传输多个字节的数据。如果数据对齐在这些边界上（如 4 字节、8 字节），CPU 可以一次性从内存中读取数据，从而提高访问效率。如果数据没有对齐，可能需要进行多次内存访问才能读取完整的数据，导致性能下降。
        - **CPU 指令集要求**：一些 CPU 指令集要求特定类型的数据（如 4 字节的 `int` 或 8 字节的 `double`）必须存储在相应的字节对齐地址上。如果数据未对齐，可能会导致硬件产生访问错误，程序崩溃。
        - **内存访问速度**：对齐的数据通常会使 CPU 以更少的指令完成内存读取操作。比如，在读取一个未对齐的数据时，CPU 可能需要读取两次内存，再进行组合，最后得到需要的数据，而对齐的数据则可以直接读取。

    - 内存碎片：内碎片外碎片
        - **外部碎片（External Fragmentation）**：
            - 这是指系统内存中虽然有足够的空闲内存总量，但这些空闲内存分布在多个不连续的小块中，无法满足大块内存的分配请求。
            - 例如，如果程序分配了多个大小不同的内存块并且随后释放了一些内存块，可能会导致内存中出现许多零散的小块空闲内存。尽管这些空闲块的总量足够大，但它们分布不连续，因此无法满足某个较大内存块的分配需求。
        - **内部碎片（Internal Fragmentation）**：
            - 内部碎片是指分配给进程的内存块比实际需要的要大，导致剩余的部分无法被有效利用。
            - 这通常发生在内存分配的粒度较大时。例如，如果系统按 4 KB 的页大小分配内存，而程序只需要 3 KB，则多出来的 1 KB 内存虽然已经被分配，但却无法被有效利用，形成内部碎片。
        - 解决方案
            - **分页机制（Paging）**：
                - Linux 使用分页机制将物理内存分成固定大小的页（通常为 4 KB）。分页可以有效减少外部碎片，因为内存分配是基于固定大小的页，而不是可变大小的内存块。即使程序需要的内存不连续，虚拟内存系统可以通过页表将这些离散的页映射为连续的虚拟地址。
                - 然而，分页机制无法完全消除内部碎片。如果进程分配的内存量不是页大小的整数倍，则最后一页中未使用的部分会形成内部碎片。
            - **伙伴系统（Buddy System）**：
                - Linux 内核使用伙伴系统来管理物理内存的分配。伙伴系统将内存分成大小为 2 的幂的块（例如 4 KB、8 KB、16 KB 等），并在需要时将这些块组合或拆分以满足内存分配请求。
                - 伙伴系统通过合并相邻的空闲块来减少外部碎片。但由于分配粒度是 2 的幂，所以仍然可能出现内部碎片。
            - **Slab 分配器**：
                - Slab 分配器是 Linux 内核中用于管理小块内存分配的系统。它通过预先分配和管理相同大小的小内存块（slab）来减少内部碎片，特别是对于频繁分配和释放的小对象。
                - Slab 分配器有效减少了小块内存分配时的内部碎片问题，并且提高了内存分配和释放的效率。

    - 信号量本质，和互斥量区别
        - **计数器**：信号量有一个整数值，这个值可以增加或减少。初始值通常设置为可用资源的数量。
        - **操作**：信号量通常支持两种基本操作：
            - **P（proberen，尝试或等待）操作**：也称为 `wait()` 或 `decrement` 操作，表示试图获取一个资源。计数器减 1。如果计数器的值为负，则进程阻塞等待。

    - 虚函数实现原理，纯虚函数怎么实现的
        - 在基类中，纯虚函数的实现部分使用 `= 0` 来声明，这告诉编译器该函数是纯虚的，基类不提供实现。
        - 纯虚函数支持多态性。当通过基类指针或引用调用纯虚函数时，将调用对象实际类型的重写版本。
        - **虚函数表（vtable）**：每个包含虚函数的类都有一个虚函数表（vtable），这个表是一个指针数组，其中每个指针指向该类中一个虚函数的实现。当类中有纯虚函数时，相应的表项会指向一个特殊的地址（通常是一个编译器生成的错误处理函数或 `nullptr`），表示该函数没有实现。
        - **虚指针（vptr）**：每个对象实例有一个指向其类的虚函数表（vtable）的指针，称为虚指针（vptr）。当通过基类指针或引用调用一个虚函数时，程序首先通过 `vptr` 指向的虚函数表（vtable）找到具体的函数地址，然后调用该函数。

        - HTTP请求过程
            1. **DNS 解析**：客户端（通常是浏览器）首先通过域名系统（DNS）将要访问的 URL 中的域名解析为服务器的 IP 地址。
            2. **建立 TCP 连接**：客户端和服务器通过 TCP 协议在传输层建立连接。客户端发送一个 TCP 三次握手请求，服务器响应，确认连接建立。
            3. **发送 HTTP 请求**：TCP 连接建立后，客户端向服务器发送一个 HTTP 请求（Request）。这个请求包含了请求方法（GET、POST 等）、请求的 URL、协议版本（如 HTTP/1.1）、请求头信息和请求体（POST 请求可能包含）。
            4. **服务器处理请求**：服务器接收到 HTTP 请求后，处理该请求（例如查询数据库、读取文件等），并准备 HTTP 响应。
            5. **发送 HTTP 响应**：服务器将处理结果作为 HTTP 响应（Response）发送回客户端。响应包含状态行（例如 `HTTP/1.1 200 OK`）、响应头信息和响应体（例如 HTML 页面、JSON 数据等）。
            6. **客户端接收并处理响应**：客户端接收到 HTTP 响应后，通常会解析响应体并显示给用户（例如渲染网页）。
            7. **关闭连接或保持连接**：如果使用的是 HTTP/1.0，服务器通常会主动关闭连接。如果使用的是 HTTP/1.1 或 HTTP/2.0，默认情况下连接会保持打开状态（长连接），以便后续请求复用同一连接。
        - HTTP是否为长连接
            - **HTTP/1.0** 默认是短连接，每次请求/响应之后连接都会关闭。如果要使用长连接，需要在请求头中添加 `Connection: keep-alive`。
            - **HTTP/1.1** 默认是长连接（Keep-Alive）。客户端和服务器会保持 TCP 连接打开，可以用于后续的请求。连接会在一定时间（通常由服务器设置的超时时间）内保持打开，直到其中一方主动关闭。
        - HTTP/2.0 是对 HTTP/1.1 的增强，旨在提高性能，减少延迟，提升 Web 应用的效率。它的主要特性包括：
            1. **二进制分帧**：HTTP/2.0 使用二进制格式而不是 HTTP/1.1 的文本格式。这种方式更高效，便于计算机处理，减少了解析的开销。
            2. **多路复用（Multiplexing）**：在一个 TCP 连接中可以同时发送多个请求和响应，消除了 HTTP/1.x 中的“队头阻塞”（Head-of-Line Blocking）问题。
            3. **请求优先级**：客户端可以为每个请求分配一个优先级，服务器根据优先级来处理请求，优化了资源的分配和使用。
            4. **数据压缩**：HTTP/2.0 对请求和响应的头信息进行 HPACK 压缩，减少了传输的数据量。
            5. **服务器推送（Server Push）**：服务器可以主动向客户端推送资源，而无需客户端明确请求。这有助于减少延迟，提升用户体验。

# We'll implement the formula for E(n) and calculate it for a given n.

def expected_burn_time(n):
    # Base case: if the length is 1, it burns out immediately.
    if n == 1:
        return 0
    
    # Recursive case
    total_time = 0
    for k in range(1, n):  # k ranges from 1 to n-1
        burn_time = min(k, n - k)
        remaining_candle_length = max(k, n - k)
        
        if remaining_candle_length >= 2:
            total_time += burn_time + expected_burn_time(remaining_candle_length)
        else:
            total_time += burn_time

    # The expected time is the average over all possible splits.
    return total_time / (n - 1)
