## Problem Statement

用户需要一个高效的邮件管理工具，当前系统存在以下问题：

1. **展示内容零散** — 无法查看邮件原文
2. **界面不友好** — 热门话题、今日话题等分散注意力
3. **缺乏状态追踪** — 无法区分已读/未读邮件
4. **无自动化** — 需要手动同步，旧邮件不会自动清理

## Goals

- 单列收件箱界面，以邮件为核心
- 支持查看原文（点击展开/收起）
- 追踪 Gmail 未读状态 + 系统处理状态
- 自动同步最近 N 天邮件
- 自动清理过期邮件

## Non-Goals

- 项目发现功能（暂不保留）
- 话题追踪功能（暂不保留）
- 日报生成功能（暂不保留）
- 发送邮件功能

## Data Model

### messages 表（全新设计）

```sql
CREATE TABLE IF NOT EXISTS messages (
    id           TEXT PRIMARY KEY,      -- Gmail message ID
    thread_id    TEXT NOT NULL,
    subject      TEXT,
    sender       TEXT,
    sender_email TEXT,                  -- 提取出的邮箱地址
    received_at  TEXT NOT NULL,         -- ISO-8601 UTC
    body_text    TEXT,                  -- 邮件原文（存储期内保留）
    msg_state    TEXT NOT NULL DEFAULT 'unread',  -- 'unread', 'read', 'starred'
    relevance    TEXT,                  -- 'high', 'medium', 'low', 'skip'
    summary_json TEXT,                  -- AI 摘要
    created_at   TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%SZ','now'))
);

CREATE INDEX idx_messages_received ON messages(received_at);
CREATE INDEX idx_messages_state ON messages(msg_state);
CREATE INDEX idx_messages_relevance ON messages(relevance);
```

**msg_state 状态：**
- `'unread'` — Gmail 未读，系统未处理
- `'read'` — Gmail 已读 或 系统已处理（两者之一即标记为 read）
- `'starred'` — Gmail 标星

**relevance 字段：**
- 由 AI 在生成摘要时自动判断
- 值：`'high'`（高重要性）、`'medium'`（中等）、`'low'`（低）、`'skip'`（跳过，非技术邮件）
- 判断依据：发件人重要性、主题相关性、内容技术深度

**summary_json 结构：**
```json
{
  "title": "邮件标题",
  "summary": "一句话摘要（100字以内）",
  "key_points": ["要点1", "要点2"],
  "topics": ["关键词1", "关键词2"],
  "relevance": "high",
  "sender_org": "发件人组织"
}

### settings 表

```sql
CREATE TABLE IF NOT EXISTS settings (
    key TEXT PRIMARY KEY,
    value TEXT NOT NULL
);
```

**默认设置：**
- `retention_days`: 7（范围：1-365）
- `auto_sync_interval_minutes`: 30（范围：5-1440）
- `last_history_id`: 空（首次同步后填充）

**请求体：**

```json

## Frontend Design

### 收件箱主页面

**布局：**
- 顶部：标题 + 设置按钮 + 同步按钮

**排序逻辑：**
1. 未读邮件（msg_state = 'unread'）

- 默认显示摘要
- 点击「展开原文」按钮，显示 body_text
- 点击「收起原文」按钮，隐藏原文

### 设置页面

**可配置项：**
- 邮件保留天数（retention_days）

## Background Tasks

### 启动流程

```
Server 启动
    │
    ├── 1. 初始化数据库
    │
    ├── 2. 启动后台同步任务
    │       │
    │       ├── 拉取最近 N 天邮件
    │       ├── 同步 msg_state
    │       └── 对新邮件调用 AI 摘要
    │
    └── 3. 启动定时清理任务
            │
            └── 每日凌晨删除过期邮件
```

### 增量同步

```python
def sync_messages():
    last_history_id = get_setting('last_history_id')

    # 尝试增量同步
    if last_history_id:
        try:
            new_messages = gmail_api.fetch_since(last_history_id)
            return process_messages(new_messages)
        except HistoryIdTooOldError:
            log.warning("sync.history_id_expired")

    # 回退到全量同步最近 N 天
    days = get_setting('retention_days', default=7)
    new_messages = gmail_api.fetch_recent(days=days)
    return process_messages(new_messages)

def process_messages(messages):
    for msg in messages:
        save_message(msg)
        summary = llm_summarize(msg.body_text)
        save_summary(msg.id, summary)

    # 同步未读状态
    unread_ids = gmail_api.get_unread_ids()
    update_msg_state(unread_ids)

    # 更新 history_id
    save_setting('last_history_id', gmail_api.get_current_history_id())
```

**LLM 配置：**
- 提供商：通过 `~/.env` 配置（支持 OpenAI、Azure、本地模型）
- 模型：从环境变量 `LLM_MODEL` 读取，默认 `gpt-4o-mini`
- Token 限制：单封邮件正文截断至 8000 字符，摘要输出限制 500 字符
- 超时处理：30 秒超时，失败后重试 2 次，仍失败则 `summary_json` 为空

### 自动清理

```python
def cleanup_old_messages():
    retention_days = get_setting('retention_days', default=7)
    cutoff_date = today - timedelta(days=retention_days)

    deleted = db.execute(
        "DELETE FROM messages WHERE received_at < ?",
        (cutoff_date,)
    )
```

### 后台任务框架

使用 FastAPI lifespan + asyncio 实现，无需额外依赖：

```python
from contextlib import asynccontextmanager

@asynccontextmanager
async def lifespan(app: FastAPI):
    # 启动时执行一次同步
    await run_initial_sync()

    # 启动后台任务
    sync_task = asyncio.create_task(sync_scheduler())
    cleanup_task = asyncio.create_task(cleanup_scheduler())

    yield

    # 关闭时取消任务
    sync_task.cancel()
    cleanup_task.cancel()

async def sync_scheduler():
    """每 N 分钟执行一次同步"""
    while True:
        interval = get_setting('auto_sync_interval_minutes', default=30)
        await asyncio.sleep(interval * 60)
        await sync_messages()

async def cleanup_scheduler():
    """每日凌晨执行清理"""
    while True:
        await asyncio.sleep(seconds_until_midnight())
        cleanup_old_messages()
```

## Implementation Phases

### Phase 1: 数据模型重构
- 重写 `models.py` schema
- 更新 `repository.py` 数据访问层
- **数据迁移策略**：
  - 备份现有数据库：`cp my_email.db my_email.db.backup`
  - 删除所有旧表（messages, summaries, digests, topic_*, projects 等）
  - 创建新表结构
  - 首次启动时从 Gmail 重新同步

### Phase 2: Gmail 同步重构
- 修改 `sync.py` 支持 `msg_state` 同步
- 增量同步逻辑优化
- **大邮件处理**：正文截断至 8000 字符，超出部分丢弃

### Phase 3: 后端 API
- 重写 `app.py` 路由
- 实现邮件列表、详情、状态更新 API
- 实现设置 API

### Phase 4: 前端界面
- 创建 `inbox.html.j2` 收件箱模板
- 创建 `settings.html.j2` 设置模板
- JavaScript 交互（展开/收起原文）

### Phase 5: 后台任务
- 实现 `auto_sync.py` 自动同步
- 实现 `cleanup.py` 自动清理
- 集成到 FastAPI lifespan

## Risks & Mitigations

| 风险 | 缓解措施 |
|------|----------|
| Gmail API 限流 | 使用增量同步，避免全量拉取 |
| AI 摘要超时 | 异步处理，先存原文后补摘要 |
| 数据丢失 | 保留 Gmail 原始邮件，只删除本地 |
